## 🆕 botnored — MapleStory v83 封包類別結構 (botnored 2022, 2026-03-28 再次確認)

**URL**: https://botnored.weebly.com/blog/maplestory-classes-v83

### 核心原則（重要）

> **"The first packet sent / received will always be in plain-text (not encrypted)."**
> **"MapleStory packets are performed though a TCP socket."**

### 封包格式原則

> "The corresponding number would be written into the packet: Actions that are handled by packets on MapleStory are defined by an identifier called an Operation Code (opcode for short), which is the first 2 bytes of the packet (a short)"

**位元組順序**：
> "Because data types are represented in little endian, the logical ordering of bytes appears reversed in comparison to typical representation of hexadecimal."

### 客戶端主要類別

| 類別 | 用途 |
|------|------|
| `CWvsContext` | 世界狀態管理（角色資料、伺服器列表） |
| `CClientSocket` | 網路連接管理，封包收發 |
| `CField` | 地圖/遊戲場景狀態管理 |
| `CLogin` | 登入狀態機管理 |

### 封包流向
```
CClientSocket::ProcessPacket()
    → 根據 opcode 分發到各 handler
    → CField::OnPacket() / CLogin::OnPacket()
    → 處理具體 Opcode
```

### MapleStory v83 封包特性總結

| 特性 | 說明 |
|------|------|
| 封包傳輸 | TCP Socket |
| 位元組順序 | Little Endian |
| Opcode 大小 | 2 bytes |
| 第一個封包 | 永遠純文字（未加密） |
| 主要加密 | AES-OFB + MapleCrypto |

## 🆕 MapleRef Wiki — Packet & Opcode 結構參考 (mapleref.fandom.com, 2026-03-28)

**URL**: https://mapleref.fandom.com/wiki/Packets
**URL**: https://mapleref.fandom.com/wiki/Operation_Code

### Packet Header 格式

> "Packet headers are 4 bytes long, and are generated with the use of the send IV, packet length (without header) and the version of MapleStory."
> "**Packet opcodes are usually 2 bytes long**."

### Opcode 位元組歷史

> "In earlier versions, the Opcode was the first byte of the packet. However, **the Opcode was changed to a short (2 bytes long)**. This changed the maximum amount of Opcodes potentially available from 255 (2^8) to 65,536 (2^16)."

### Send vs Recv Opcode 區分（重要）

> "Transmitting from the server and receiving on the client has one set of agreed upon Opcodes, where transmitting from the client and receiving on the server has yet another."

| 方向 | Opcode Set | 說明 |
|------|-----------|------|
| **S→C (Send)** | Server Opcodes | 伺服器發送到客戶端 |
| **C→S (Recv)** | Client Opcodes | 客戶端發送到伺服器 |

**重要**：同一個數字的 Opcode，在不同方向代表**完全不同**的含義！

### MapleCrypto Header 格式
```
[2 bytes: 封包長度][2 bytes: 混淆後的 Opcode]
```

### AES 加密特性
> "Each Maplestory connection session uses two ciphers, each having a state containing an IV (Initialization Vector) which is passed to AES-OFB. The Encryption key and IV are used to encrypt packets (data transmitted to and from the server)."

### 狀態機模式

```javascript
function start() {
    status = -1;
    action(1, 0, 0);
}

function action(mode, type, selection) {
    if (mode == -1) {
        cm.dispose();  // 結束對話
    } else {
        if (mode == 0 && status == 0) {
            cm.dispose();  // 結束對話
            return;
        }
        if (mode == 1) {
            status++;  // 下一頁
        } else {
            status--;  // 上一頁
        }
        
        if (status == 0) {
            cm.sendNext("Hello! How can I help you?");
        } else if (status == 1) {
            // 處理選項
            cm.dispose();
        }
    }
}
```

## 🆕 CClientSocket::ProcessPacket 封包分發架構 (2026-03-28 總結)

### 所有客戶端接收封包的入口點

```
CClientSocket::ProcessPacket
    ↓
根據 MapleCrypto header 解密
    ↓
提取 opcode (2 bytes, little endian)
    ↓
switch(opcode) 分發
    ↓
CLogin::OnPacket()    // 登入畫面
CField::OnPacket()    // 遊戲中
CCashShop::OnPacket() // 現金商店
CWvsContext::OnPacket() // warp 前
```

### 為什麼知道 Opcode 就能找到處理函數？

- `CClientSocket::ProcessPacket` 內有大 switch 表
- 每個 case 對應一個 opcode
- case 內直接調用具體的 OnPacket 處理函數
- 知道 opcode → IDA 中快速定位函數

### IDA 分析 Opcode 的價值

| 用途 | IDA 能做到 |
|------|-----------|
| 理解客戶端內部運作 | ✅ |
| 修改客戶端功能 | ✅ |
| 對照開源源碼確認差異 | ✅ |
| 開發新工具（如 MaplePE） | ✅ |

## 🆕 WZ 修改完整流程 (2026-03-28 總結)

### WZ 編輯黃金法則

> **"DO NOT USE the server's XMLs for reimporting into the client's WZ, it WILL generate bugs."**

### 正確 WZ 編輯流程

```
1. HaRepacker 4.2.4 開啟 WZ（使用 GMS old 加密）
2. 使用 UI 直接修改數值（禁止刪除/重新導入節點！）
3. 儲存 WZ
4. 在 HaRepacker 中選擇 "Private Server..." 導出
5. 將導出的 XML 覆蓋到伺服器的 WZ.XML 目錄
```

### 為什麼不能直接用伺服器的 XML？

| 問題 | 原因 |
|------|------|
| 客戶端 WZ 加密不同 | 伺服器 XML 是解密後格式 |
| 重新導入可能破壞結構 | 刪除/導入節點會造成數據丟失 |
| Opcode 不同步 | 客戶端和伺服器的 opcode 可能不匹配 |

### 完整工作流程
```
Step 1: IDA Pro 反編譯
  → 在 IDA 中找到 CField::OnPacket 或 CLogin::OnPacket
  → 產生 pseudocode (.p64 或 .txt)

Step 2: 複製 pseudocode
  → 在 Functions/ 目錄建立 .txt 檔案
  → 貼上 IDA 產生的 C-style pseudocode

Step 3: Python 分析
  → cd MaplePacketPuller/IDA\ Maple\ Script/src/main/python
  → python main.py
  → 輸入分析類型（完整/標準）
  → 輸入要分析的檔案名稱

Step 4: 輸出結果
  → 封包結構自動寫入 FuncOutput/ 目錄
  → 控制台顯示 Decode1/Decode2/Decode4/DecodeStr 等欄位
```

### SpiritIDAPlugin 功能亮點
- 直接在 IDA 中格式化 pseudocode 為伺服器 encode 友好格式
- 支援自訂準確度選項
- 自動搜尋 InHeader opcode
- ⚠️ 限制：while() 迴圈處理不完善（建議設 GET_ALL_DECODES=true）
