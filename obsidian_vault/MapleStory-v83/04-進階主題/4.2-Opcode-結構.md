# 4.2 Opcode 結構分析

## 什麼是 Opcode？
Opcode (Operation Code) 是網路封包中用於標識操作類型的唯一識別符。

## v83 Opcode 特性
- **已高度文檔化**：v83 的 Opcode 幾乎全部被映射
- **使用 AES 加密**：需配合客戶端金鑰
- **高度客製化**：每個私服可能使用不同設定

## Opcode 結構
```
[封包頭部][Opcode (2-4 bytes)][資料...]
```

### 封包頭部格式
```
偏移  長度   內容
0x00  4      MapleCrypto 混淆頭部
      (包含長度 + 混淆 opcode)
```

### v83 Opcode 格式
- Opcode 為 **2 bytes** (little endian)
- 例如：Opcode `0x0001` 在封包中顯示為 `01 00`

## 🔑 MapleCrypto 頭部混淆機制

### 混淆原理
```
原始 opcode: 0x0001 (例如 LOGIN_PASSWORD)
       IV: [0x7A, 0x52, ...]

Step 1: XOR + SHIFT
  verify = (opcode ^ iv[0]) + (iv[1] << 8)
        = (0x0001 ^ 0x7A) + (0x52 << 8)
        = 0x007B + 0x5200
        = 0x527B

Step 2: 位旋轉 (rotate left 3 bits)
  verify = (verify << 3) | (verify >> 13)
        = 0x527B << 3 | 0x527B >> 13
        = 0x29BD8 | 0x00
        = 0x29BD8

Step 3: 取低位 16 bits
  verify = 0x29BD8 & 0xFFFF = 0x9BD8
```

## 🔐 封包加密機制（v83）

### 雙層加密架構
v83 客戶端使用兩層加密：

```
客戶端發送流程：
1. 原始封包: [Opcode][Data...]
2. Shanda 解密/加密 (對稱式)
3. AES 解密 (CBC 模式)
4. 傳送

伺服器接收流程：
1. 接收加密封包
2. AES 解密
3. Shanda 解密
4. 處理
```

### 第一個封包（明文）
- MapleStory 連線時，**第一個封包是明文**（未加密）
- 之後所有封包都經過 Shanda + AES 雙重加密
- 這是客戶端與伺服器交換加密金鑰（IV）的時機

### Shanda 加密
- 早期 MapleStory 使用的對稱加密演算法
- 每個版本的金鑰不同
- v83 使用特定的金鑰陣列

### AES 加密
- 使用 CBC 模式
- 金鑰和 IV 在客戶端與伺服器間協商
- 伺服器端在 ServerConstants 中定義

### 登入封包範例
客戶端發送的登入封包結構：
```
[E2 FB] [71 1A 5B 6A...] <- 經 Shanda + AES 加密後
解密後应为: [01 00] [帳號][密碼]...
```
其中 `01 00` 是 LOGIN_PASSWORD opcode (little endian)

### 封包加密移除（自架伺服器）
如果想在本地客戶端移除加密：
- 使用 IDA 分析，找出 AES/Shanda 加密函數
- NOP 掉這些函數的調用
- 伺服器端也需要同步修改以發送未加密封包

### CRC32 GameGuard 檢查
- 客戶端有一個舊版 GameGuard 的 CRC32 校驗
- 當發送封包時，會附加一個 opcode (0x13/19)
- 這在私服環境中無實際作用，可忽略

## 🆕 MapleCrypto Header 加密/解密公式 (v83 核心)

### 加密端（伺服器/客戶端發送）
```java
public byte[] getPacketHeader(int opcode, int length, int iv) {
    int verify = (opcode ^ (iv & 0xFF)) + ((iv >> 8) & 0xFF) * 0x100;
    verify = (verify << 3) | (verify >>> 5);
    verify &= 0xFFFF;
    return new byte[] {
        (byte) (verify & 0xFF),
        (byte) ((verify >>> 8) & 0xFF),
        (byte) (length & 0xFF),
        (byte) ((length >>> 8) & 0xFF)
    };
}
```

### 解密端（接收方）
```java
public int retrieveOpcode(byte[] header, int iv) {
    int verify = (header[0] & 0xFF) | ((header[1] << 8) & 0xFF00);
    verify = (verify >>> 3) | ((verify << 5) & 0xFFFF);
    return (verify - ((iv >> 8) & 0xFF) * 0x100) ^ (iv & 0xFF);
}
```

## 🆕 CMS (中國瑪奇) 登入封包解密實戰案例 (RaGEZONE, 2018)

**來源**：https://forum.ragezone.com/threads/help-remove-encrypt-decrypt-in-client-cms-chinese-maplestory.1155998/

這是 CMS v83 的登入封包三層解密流程實際案例：

### 客戶端發送原始密文 (C→S)
```
64 A4 9F 07 FD 4D 6C 49 4A 3D 96 FE C8 44 85 E4 79 FE EE E3 C7 3E 9B BF 42 3B 97 B2 4E 36 F8 DB F4 8C 0D D9
```

### 第一步：AES-OFB 解密（使用版本決定的 Key 和 IV）
```
解密後：71 1A 5B 6A 7E B9 C9 EA 15 17 7E E4 C0 B4 64 6D 1E E7 8A 13 C9 B0 8C 09 4C 7F F5 6F F8 A9 2C 5D 03 2D D4 D5
```

### 第二步：Shanda Decryption（對稱加密）
```
解密後：E2 FB 40 7E 67 08 B6 77 F4 CC 0E 3E 5D 72 D1 9A 2C 7C F6 44 16 6A 7F 2F ED 39 C8 DE 96 F7 95 04 59 88 C9 6F
```

### 問題分析
```
解密後第一個 opcode 為 E2 FB（Big Endian）→ 實際 opcode = 0xFBE2
但正常 LOGIN_PASSWORD opcode 應該是 0x01 00
→ 原因：版本不匹配，或 IV 金鑰不正確
→ 這說明解密流程有問題，或 AES Key 版本不對
```

### 三層加密流程圖
```
發送端：
明文資料 → AES-OFB 加密 → Shanda 加密 → 網路傳輸

接收端：
網路接收 → Shanda 解密 → AES-OFB 解密 → 明文資料
```

### v83 加密特點
| 加密層 | 演算法 | 說明 |
|--------|--------|------|
| 第一層 | MapleCrypto (頭部) | 4-byte header，含混淆 opcode |
| 第二層 | Shanda Encryption | v83 仍使用，已於 v118+ 移除 |
| 第三層 | AES-OFB | 256-bit key，版本決定的靜態金鑰 |

> 💡 **重要教訓**：若解密出來的 opcode 不對，優先檢查 AES Key 版本是否匹配！

### v83 封包加密三層架構
```
┌─────────────────────────────────────────────────────┐
│ Layer 1: MapleCrypto (頭部混淆)                     │
│   - 4 bytes header (混淆 opcode + length)           │
│   - 防止靜態分析 opcode                             │
│                                                        │
│ Layer 2: Shanda Encryption (對稱加密)                │
│   - 早期版本使用                                     │
│   - 已於 v118+ 移除                                 │
│                                                        │
│ Layer 3: AES-OFB (主加密)                           │
│   - v83 主要加密方式                                │
│   - 256-bit key, 4-byte IV                          │
│   - 版本決定的靜態金鑰                              │
└─────────────────────────────────────────────────────┘
```

## 🆕 CInPacket 解碼流程深度解析 (RaGEZONE, 2016)

**來源**: https://forum.ragezone.com/threads/the-java-flow-of-maple-story-functions.1113547/

**封包解碼流程**：
```
1. CClientSocket 接收 TCP 資料
2. CInPacket::Decode() 被調用
   - 解析 MapleCrypto header (4 bytes)
   - 解密並提取 opcode (2 bytes)
   - 剩餘為封包內容 (payload)
3. CInPacket 構造完成 (nType = opcode, buffer = payload)
4. 根據 nType 分發到具體處理函數
5. 在處理函數中讀取 buffer 的各個欄位
```

### IDA 中對應的 Decode 函數

| IDA 函數名 | MaplePacketWriter 方法 | 說明 |
|-----------|----------------------|------|
| `Decode1` | `mplew.write` | 讀取 1 byte |
| `Decode2` | `mplew.writeShort` | 讀取 2 bytes (short/opcode) |
| `Decode4` | `mplew.writeInt` | 讀取 4 bytes (int) |

- 每個 case 對應一個 opcode
- case 內直接調用具體的 OnPacket 處理函數
- 因此知道 opcode 就能在 IDA 中快速定位函數

### CInPacket 封包接收完整流程
```cpp
// 客戶端接收封包 (IDA pseudocode)
void __thiscall CClientSocket::ProcessPacket(CClientSocket *this) {
    while (true) {
        // 讀取 4-byte header
        BYTE header[4];
        recv(socket, header, 4);
        
        // MapleCrypto 解混淆
        int packetLen = MapleCrypto::decryptLength(header, ivRecv);
        int opcode = MapleCrypto::decryptOpcode(header, ivRecv);
        
        // 分配 buffer
        byte *buffer = new byte[packetLen];
        
        // 接收剩餘資料 (packetLen bytes)
        recv(socket, buffer, packetLen);
        
        // AES-OFB + Shanda 解密
        MapleCrypto::decrypt(buffer, packetLen, ivRecv);
        
        // 構造 CInPacket
        CInPacket packet;
        packet.nType = opcode;      // 設定 opcode
        packet.buffer = buffer;      // 設定 payload
        
        // 分發到處理函數
        switch (opcode) {
            case 0x00: CLogin::OnPacket(&packet); break;
            case 0x01: CLogin::OnLoginPassword(&packet); break;
            // ...
        }
    }
}
```

### MapleCrypto 解密長度演算法
```java
public static int decryptLength(byte[] header, int iv) {
    int length = (header[2] & 0xFF) | ((header[3] << 8) & 0xFF00);
    length ^= (iv >> 16) & 0xFFFF;
    return length;
}
```

SpiritIDAPlugin 是 MaplePacketPuller 的 IDA Pro 插件版，可直接在 IDA 中提取封包結構。

**支援版本**:
| 環境 | 支援狀態 |
|------|---------|
| Python 3.8.5 | ✅ 測試通過 |
| Python 3.6.12 & 3.8.6 | ✅ 測試通過 |
| IDA Pro 7.0 (32-bit) | ✅ 測試通過 |
| IDA Pro 7.5 | ✅ 測試通過 |
| IDA Pro 6.8 | ✅ 測試通過 |
| Python 2.7 | ❌ 不相容 (os.scandir) |

### 完整工作流程
```
Step 1: IDA Pro 反編譯
  → 在 IDA 中找到 CField::OnPacket 或 CLogin::OnPacket
  → 產生 pseudocode (.p64 或 .txt)

Step 2: 複製 pseudocode
  → 在 Functions/ 目錄建立 .txt 檔案
  → 貼上 IDA 產生的 C-style pseudocode

Step 3: Python 分析
  → cd MaplePacketPuller/IDA\ Maple\ Script/src/main/python
  → python main.py
  → 輸入分析類型（完整/標準）
  → 輸入要分析的檔案名稱

Step 4: 輸出結果
  → 封包結構自動寫入 FuncOutput/ 目錄
  → 控制台顯示 Decode1/Decode2/Decode4/DecodeStr 等欄位
```

### SpiritIDAPlugin 自動化亮點
- 直接在 IDA 中格式化 pseudocode 為伺服器 encode 友好格式
- 支援自訂準確度選項
- 自動搜尋 InHeader opcode
- ⚠️ 限制：while() 迴圈處理不完善（建議設 GET_ALL_DECODES=true）

### MaplePacketPuller 產出範例
```
輸入 (IDA pseudocode):
int __thiscall sub_ABC1230(CInPacket *this)
{
  CInPacket::Decode4(this);   // opcode
  CInPacket::DecodeStr(this); // username
  CInPacket::DecodeStr(this); // password
  ...
}

輸出 (封包結構):
LOGIN_PASSWORD (Opcode: 0x01)
  - int opcode
  - string username
  - string password
```
