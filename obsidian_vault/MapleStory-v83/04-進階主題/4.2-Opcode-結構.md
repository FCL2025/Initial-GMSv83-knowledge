# 4.2 Opcode 結構分析

## 什麼是 Opcode？
Opcode (Operation Code) 是網路封包中用於標識操作類型的唯一識別符。

## v83 Opcode 特性
- **已高度文檔化**：v83 的 Opcode 幾乎全部被映射
- **使用 AES 加密**：需配合客戶端金鑰
- **高度客製化**：每個私服可能使用不同設定

## Opcode 結構
```
[封包頭部][Opcode (2-4 bytes)][資料...]
```

### 封包頭部格式
```
偏移  長度   內容
0x00  4      MapleCrypto 混淆頭部
      (包含長度 + 混淆 opcode)
```

### v83 Opcode 格式
- Opcode 為 **2 bytes** (little endian)
- 例如：Opcode `0x0001` 在封包中顯示為 `01 00`

## 🔑 MapleCrypto 頭部混淆機制

### 混淆原理
```
原始 opcode: 0x0001 (例如 LOGIN_PASSWORD)
       IV: [0x7A, 0x52, ...]

Step 1: XOR + SHIFT
  verify = (opcode ^ iv[0]) + (iv[1] << 8)
        = (0x0001 ^ 0x7A) + (0x52 << 8)
        = 0x007B + 0x5200
        = 0x527B

Step 2: 位旋轉 (rotate left 3 bits)
  verify = (verify << 3) | (verify >> 13)
        = 0x527B << 3 | 0x527B >> 13
        = 0x29BD8 | 0x00
        = 0x29BD8

Step 3: 取低位 16 bits
  verify = 0x29BD8 & 0xFFFF = 0x9BD8
```

## 📊 IDA 分析技巧

### 1. 找到關鍵函數
使用 IDA 分析 MapleStory.exe，搜尋：
- `CWvsContext::OnPacket`
- `SendPacket`
- `DecryptPacket`
- `MapleAESOFB` (加密類)

### 2. IDA 字串搜尋技巧
- 搜尋 `"opcode"` 或 `"Packet"` 相關字串
- 查找 StringPool 引用 (`push opcode 68`)
- AOB (Array of Bytes) 搜尋可找到 opcode 表

### 3. 分析流程
```
1. 找到 SendPacket/RecvPacket 函數
2. 分析封包加密邏輯 (MapleCrypto)
3. 提取 Opcode 表
4. 對比伺服器代碼
```

### 4. 靜態分析 vs 動態分析
- **靜態分析 (IDA)**：分析二進位碼，不執行
- **動態分析 (OllyDbg/CE)**：在執行時追蹤

## 🐾 MapleCrypto Java 實現參考
```java
// Cosmic / HeavenMS 的 MapleCrypto 實現關鍵片段
public class MapleCrypto {
    private final byte[] key;
    private final int iv;

    public byte[] getPacketHeader(int opcode) {
        int verify = (opcode ^ (iv & 0xFF)) + ((iv >> 8) & 0xFF) * 0x100;
        verify = (verify << 3) | (verify >>> 5);
        verify &= 0xFFFF;
        return new byte[] {
            (byte) (verify & 0xFF),
            (byte) ((verify >>> 8) & 0xFF),
            (byte) (length & 0xFF),
            (byte) ((length >>> 8) & 0xFF)
        };
    }
}
```

## 🆕 MapleCrypto Header 加密/解密公式 (v83 核心)

### 加密端（伺服器/客戶端發送）
```java
public byte[] getPacketHeader(int opcode, int length, int iv) {
    int verify = (opcode ^ (iv & 0xFF)) + ((iv >> 8) & 0xFF) * 0x100;
    verify = (verify << 3) | (verify >>> 5);
    verify &= 0xFFFF;
    return new byte[] {
        (byte) (verify & 0xFF),
        (byte) ((verify >>> 8) & 0xFF),
        (byte) (length & 0xFF),
        (byte) ((length >>> 8) & 0xFF)
    };
}
```

### 解密端（接收方）
```java
public int retrieveOpcode(byte[] header, int iv) {
    int verify = (header[0] & 0xFF) | ((header[1] << 8) & 0xFF00);
    verify = (verify >>> 3) | ((verify << 5) & 0xFFFF);
    return (verify - ((iv >> 8) & 0xFF) * 0x100) ^ (iv & 0xFF);
}
```

## 🆕 CMS (中國瑪奇) 登入封包解密實戰案例 (RaGEZONE, 2018)

**來源**：https://forum.ragezone.com/threads/help-remove-encrypt-decrypt-in-client-cms-chinese-maplestory.1155998/

這是 CMS v83 的登入封包三層解密流程實際案例：

### 客戶端發送原始密文 (C→S)
```
64 A4 9F 07 FD 4D 6C 49 4A 3D 96 FE C8 44 85 E4 79 FE EE E3 C7 3E 9B BF 42 3B 97 B2 4E 36 F8 DB F4 8C 0D D9
```

### 第一步：AES-OFB 解密（使用版本決定的 Key 和 IV）
```
解密後：71 1A 5B 6A 7E B9 C9 EA 15 17 7E E4 C0 B4 64 6D 1E E7 8A 13 C9 B0 8C 09 4C 7F F5 6F F8 A9 2C 5D 03 2D D4 D5
```

### 第二步：Shanda Decryption（對稱加密）
```
解密後：E2 FB 40 7E 67 08 B6 77 F4 CC 0E 3E 5D 72 D1 9A 2C 7C F6 44 16 6A 7F 2F ED 39 C8 DE 96 F7 95 04 59 88 C9 6F
```

### 問題分析
```
解密後第一個 opcode 為 E2 FB（Big Endian）→ 實際 opcode = 0xFBE2
但正常 LOGIN_PASSWORD opcode 應該是 0x01 00
→ 原因：版本不匹配，或 IV 金鑰不正確
→ 這說明解密流程有問題，或 AES Key 版本不對
```

### 三層加密流程圖
```
發送端：
明文資料 → AES-OFB 加密 → Shanda 加密 → 網路傳輸

接收端：
網路接收 → Shanda 解密 → AES-OFB 解密 → 明文資料
```

### v83 加密特點
| 加密層 | 演算法 | 說明 |
|--------|--------|------|
| 第一層 | MapleCrypto (頭部) | 4-byte header，含混淆 opcode |
| 第二層 | Shanda Encryption | v83 仍使用，已於 v118+ 移除 |
| 第三層 | AES-OFB | 256-bit key，版本決定的靜態金鑰 |

> 💡 **重要教訓**：若解密出來的 opcode 不對，優先檢查 AES Key 版本是否匹配！

### v83 封包加密三層架構
```
┌─────────────────────────────────────────────────────┐
│ Layer 1: MapleCrypto (頭部混淆)                     │
│   - 4 bytes header (混淆 opcode + length)           │
│   - 防止靜態分析 opcode                             │
│                                                        │
│ Layer 2: Shanda Encryption (對稱加密)                │
│   - 早期版本使用                                     │
│   - 已於 v118+ 移除                                 │
│                                                        │
│ Layer 3: AES-OFB (主加密)                           │
│   - v83 主要加密方式                                │
│   - 256-bit key, 4-byte IV                          │
│   - 版本決定的靜態金鑰                              │
└─────────────────────────────────────────────────────┘
```
