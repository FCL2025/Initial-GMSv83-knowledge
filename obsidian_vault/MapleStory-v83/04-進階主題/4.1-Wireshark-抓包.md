# 4.1 Wireshark 抓包教學

## 概述
使用 Wireshark 抓取 MapleStory 網路封包，分析客戶端與伺服器之間的通訊。

## Wireshark 下載
官網：https://www.wireshark.org/

## 抓包步驟

### 1. 設置過濾器
MapleStory 使用的連接埠通常為：
- 登入端口：7575
- 遊戲端口：8484 (或自定義)

### 2. 過濾表達式
```
tcp.port == 7575 || tcp.port == 8484
```

### 3. 分析重點
- **封包方向**：客戶端發送 (C→S) vs 伺服器發送 (S→C)
- **封包大小**：識別不同類型的操作
- **封包內容**：尋找可讀字串（Hello Packet 為純文字）

## 🐾 v83 封包加密架構

### 三層加密結構
```
┌─────────────────────────────────────────────────────┐
│  1. MapleCrypto (頭部混淆)                          │
│     - 防止靜態分析 opcode                           │
│     - Header: 4 bytes (長度 + 混淆 opcode)           │
│                                                        │
│  2. Shanda Encryption (IOBufferManipulator)          │
│     - Nexon 內部名稱：IOBufferManipulator            │
│     - 實作函數：IOBufferManipulator.Encrypt(pBuff)  │
│     - 在 AES 之前應用                              │
│     - 已於 v118+ 移除                               │
│                                                        │
│  3. AES-OFB (主加密)                                │
│     - v83 主要加密方式                              │
│     - 256-bit key, 4-byte IV                       │
│     - 版本決定的靜態金鑰                           │
└─────────────────────────────────────────────────────┘

加密流程：
明文 → IOBufferManipulator.Encrypt() (Shanda) → AES-OFB → MapleCrypto header → 網路
```

### 重要原則
> **第一個發送/接收的封包永遠是純文字（未加密）**

這個未加密的封包就是 **Hello Packet**，用於協商加密參數。

## 📨 Hello Packet 詳細結構

### 客戶端 → 伺服器 (C→S) Hello
```
偏移    長度   內容
0x00    2      封包總長度 (little endian)
0x02    2      Maple 版本號 (如 0x0053 = 83)
0x04    n      版本字串 (如 "83")
n+4     4      SendIV (客戶端指定的發送 IV)
n+8     4      RecvIV (客戶端指定的接收 IV)
n+12    2      地區標誌 (MAPLE_LOCALE)
n+14    2      Unknown (未知用途)
```

### 伺服器 → 客戶端 (S→C) Hello
```
偏移    長度   內容
0x00    2      封包總長度
0x02    2      Maple 版本號
0x04    n      版本字串
n+4     4      RecvIV (伺服器回應)
n+8     4      SendIV (伺服器回應)
n+12    2      地區標誌
n+14    2      Unknown
```

### MapleCrypto 頭部混淆演算法
```python
# 加密端：對 opcode 進行混淆
def maple_encrypt_header(opcode, iv):
    verify = (opcode ^ iv[0]) + (iv[1] << 8)
    verify = (verify << 3) | (verify >> 5)  # 位旋轉 3 位
    return verify & 0xFFFF

# 解密端：反向操作
def maple_decrypt_header(encrypted, iv):
    verify = encrypted
    verify = (verify >> 3) | ((verify << 5) & 0xFFFF)  # 反向旋轉
    return (verify - (iv[1] << 8)) ^ iv[0]
```

## 📡 登入流程封包序列

### 完整登入流程
```
1. C→S: Hello Packet (純文字)
   → 客戶端告知版本、IV

2. S→C: Hello Packet (純文字)
   → 伺服器確認版本、回應 IV

3. C→S: 登入請求 (已加密)
   Opcode: 0x01 (LOGIN_PASSWORD)
   Data: 帳號 + 密碼 (已 AES 加密)

4. S→C: 登入結果 (已加密)
   Opcode: 0x00 (LOGIN_STATUS)
   Data: 結果碼 + 帳號資訊

5. C→S: 世界選擇 (已加密)
   Opcode: 0x05 (SELECT_CHARLIST)

6. S→C: 角色列表 (已加密)
   Opcode: 0x03 (CHARLIST)
   Data: 角色資料陣列

7. C→S: 選擇角色進入遊戲 (已加密)
   Opcode: 0x0B (PLAYER_DROP)
```

## 🔧 Wireshark 實戰技巧

### 設定顏色規則
```
紅色 (異常):    tcp.len > 2000      (大型封包)
綠色 (客戶端):  ip.src == 127.0.0.1 (本地發送)
藍色 (伺服器):  ip.dst == 127.0.0.1 (本地接收)
黃色 (登入):    tcp.port == 7575    (登入端口)
```

### Follow TCP Stream 分析
1. 右鍵任意封包 → Follow → TCP Stream
2. 檢視完整會話的明文部分（Hello Packet）
3. 分離 C→S 和 S→C 流量
4. 識別加密封包的長度規律

### 識別加密封包
```
徵兆：
- 封包長度固定 (如每次都是 48 bytes)
- 內容看起來像隨機位元組
- 相同操作產生的封包每次都不同 (因為 IV 變化)
```

## 🆕 完整登入流程封包序列 (v83 實戰)

**研究時間**：2026-03-24

### 完整 10 步登入封包序列
```
1. C→S Hello Packet (plaintext - 無 opcode header)
   - 告知版本、SendIV、RecvIV
   - 為純文字，Wireshark 直接可見

2. S→C Hello Packet (plaintext)
   - 伺服器確認版本、回應 IV

3. C→S LOGIN_PASSWORD (encrypted)
   Opcode: 0x01
   Data: 帳號字串 + 密碼字串 (AES-OFB 加密)

4. S→C LOGIN_STATUS (encrypted)
   Opcode: 0x00
   Data: 結果碼 + 帳號資訊

5. C→S SERVERLIST_REQUEST (encrypted)
   Opcode: 0x06
   - 請求世界伺服器列表

6. S→C SERVERLIST (encrypted)
   Opcode: 0x0C
   Data: 世界伺服器名稱、負載、人數上限

7. C→S CHARLIST_REQUEST (encrypted)
   Opcode: 0x07
   - 請求指定世界的角色列表

8. S→C CHARLIST (encrypted)
   Opcode: 0x03
   Data: 角色資料陣列（名稱、等級、職業、外觀）

9. C→S CHAR_SELECT (encrypted)
   Opcode: 0x08
   - 選擇角色

10. C→S PLAYER_DROP (encrypted)
    Opcode: 0x0B
    - 進入遊戲世界
```

### v83 登入結果碼（Login Status Code）

| 結果碼 | 意義 |
|--------|------|
| 0 | 登入成功 |
| 5 | 帳號或密碼錯誤 |
| 7 | 帳號已停用 |
| 13 | 刪除中（冷靜期） |
| 23 | 登入太頻繁（防機器人延遲） |
| 其他 | 未知錯誤 |

> 來源：RaGEZONE + 實戰分析

### 為什麼第一個封包是純文字？
- MapleStory 使用「加密握手」流程
- 第一個封包用於客戶端和伺服器**協商 IV（初始化向量）**
- 雙方需要用相同的 IV 初始化 AES-OFB 加密器
- 若加密，雙方都無法解密對方的第一個訊息

### Wireshark 分析技巧
- 使用 `tcp.port == 7575 || tcp.port == 8484` 過濾
- 在 Hello Packet 中確認版本號（如 0x0053 = 83）
- Follow TCP Stream 查看完整明文握手
- 加密封包長度固定（可用於識別類型）

## 🆕 CClientSocket::ProcessPacket 封包流向（RaGEZONE, 2019）

**來源**: https://forum.ragezone.com/threads/how-does-one-find-a-structure-of-a-packet-via-ida.1162654/

**所有客戶端接收封包的入口點**：
```
CClientSocket::ProcessPacket()
    ↓
根據 MapleCrypto header 解密
    ↓
提取 opcode (2 bytes, little endian)
    ↓
switch(opcode) 分發
    ↓
CLogin::OnPacket()    // 登入畫面
    或
CField::OnPacket()    // 遊戲中
```

**為什麼知道 Opcode 就能找到處理函數**：
- `CClientSocket::ProcessPacket` 內有一個大 switch 表
- 每個 case 對應一個 opcode
- case 內直接調用具體的 OnPacket 處理函數
- 因此知道 opcode 就能在 IDA 中快速定位函數

**封包處理函數命名慣例**：
| 函數前綴 | 狀態 | 說明 |
|---------|------|------|
| `CLogin::On*` | 登入畫面 | 登入相關封包處理 |
| `CField::On*` | 遊戲中 | 遊戲內封包處理 |
| `CShop::On*` | 商場 | 商場相關封包處理 |

SpiritIDAPlugin 是 MaplePacketPuller 的 IDA Pro 插件版，可直接在 IDA 中提取封包結構。

**支援版本**:
| 環境 | 支援狀態 |

| IDA Pro 7.5 | ✅ 測試通過 |
| IDA Pro 6.8 | ✅ 測試通過 |
| Python 2.7 | ❌ 不相容 (os.scandir) |

### 完整工作流程
```
Step 1: IDA Pro 反編譯
  → 在 IDA 中找到 CField::OnPacket 或 CLogin::OnPacket
  → 產生 pseudocode (.p64 或 .txt)

Step 2: 複製 pseudocode
  → 在 Functions/ 目錄建立 .txt 檔案
  → 貼上 IDA 產生的 C-style pseudocode

Step 3: Python 分析
  → cd MaplePacketPuller/IDA\ Maple\ Script/src/main/python
  → python main.py
  → 輸入分析類型（完整/標準）
  → 輸入要分析的檔案名稱

Step 4: 輸出結果
  → 封包結構自動寫入 FuncOutput/ 目錄
  → 控制台顯示 Decode1/Decode2/Decode4/DecodeStr 等欄位
```

### SpiritIDAPlugin 自動化亮點
- 直接在 IDA 中格式化 pseudocode 為伺服器 encode 友好格式
- 支援自訂準確度選項
- 自動搜尋 InHeader opcode
- ⚠️ 限制：while() 迴圈處理不完善（建議設 GET_ALL_DECODES=true）

### MaplePacketPuller 產出範例
```
輸入 (IDA pseudocode):
int __thiscall sub_ABC1230(CInPacket *this)
{
  CInPacket::Decode4(this);   // opcode
  CInPacket::DecodeStr(this); // username
  CInPacket::DecodeStr(this); // password
  ...
}

輸出 (封包結構):
LOGIN_PASSWORD (Opcode: 0x01)
  - int opcode
  - string username
  - string password
```

## 🆕 v83 AES Key 實戰參考 (CMS 中國伺服器)

**來源**: https://forum.ragezone.com/threads/help-remove-encrypt-decrypt-in-client-cms-chinese-maplestory.1155998/

### v83 AES Key 參考 (CMS/Chinese MS v83)
```java
// AES UserKey (first column per row)
0x13, 0x08, 0x06, 0xB4, 0x1B, 0x0F, 0x33, 0x52

// IV Transformation Key
0xC65053F2

// 客戶端版本
0x23 (35 decimal)
```

### MapleCrypto + Shanda + AES 三層解密流程圖
```
發送端 (C→S):
明文資料
  → AES-OFB 加密 (使用版本 Key + IV)
  → Shanda 加密 (對稱加密)
  → MapleCrypto 頭部混淆 (4-byte header)
  → 網路傳輸

接收端 (S→C):
網路接收
  → MapleCrypto 解混淆 (提取 opcode + length)
  → Shanda 解密
  → AES-OFB 解密
  → 明文資料
```

### 排查解密失敗的檢查清單
| 檢查項目 | 正確值 | 症狀 |
|---------|--------|------|
| AES Key | 版本決定的 32-byte key | 解密出乱码 |
| IV | Hello Packet 協商的 IV | opcode 不匹配 |
| Shanda Key | 版本特定的對稱金鑰 | 資料順序錯誤 |
| 版本號 | 0x0053 (83) | Hello Packet 失敗 |
| Opcode 表 | 需與客戶端匹配 | switch case 走错分支 |

| 徵兆 | 封包類型 |

## 🆕 MapleRef Wiki 封包結構參考 (2026-03-25 新增)

**來源**: https://mapleref.fandom.com/wiki/Packets

### Opcode 核心概念
> "Actions that are defined by an identifier called an Operation Code (opcode for short), which is the first 2 bytes of the packet (a short)"

### Send vs Receive Opcode 區分
| 方向 | 說明 |
|------|------|
| **S→C (Send)** | 伺服器發送，客戶端接收 |
| **C→S (Recv)** | 客戶端發送，伺服器接收 |

**重要**：同一個 Opcode 數字可能代表不同含義，視方向而定！

### MapleRef 使用方式
```
1. 前往 https://mapleref.fandom.com/wiki/Packets
2. 選擇目標版本 (v83)
3. 查看 Send (S→C) 或 Recv (C→S) 封包結構
4. 配合 Wireshark 抓包分析
```

## 🆕 v83 商場封包 Dupe 分析 (RaGEZONE, 2026-03-25 新增)

**來源**: https://pdfcoffee.com/v83-dupe-pdf-free.html

### Opcode 0x6F (SHOP_BUY) 結構
```
6F 00 20 00 FF FF FF 80 00
```
- `6F 00`: Opcode (little endian = 0x006F)
- `20 00`: 商店ID (0x0020 = 32)
- `FF FF FF 80`: 物品索引 + 最大 Mesos 標誌
- `00`: 結束標記

## 🆕 MapleStory 封包加密核心原理 - MapleCrypto 雙金鑰機制 (2026-03)

**來源**: https://pkg.go.dev/github.com/defaultmagi/maplelib

### 核心概念
> MapleStory uses two keys: one is used to encrypt sent packets and the other is used to decrypt received packets. This is valid for both the client and the server.

### 四個 IV 的意義
| IV | 擁有者 | 用途 |
|----|--------|------|
| 客戶端 SendIV | 客戶端 | 加密發送到伺服器的封包 |
| 客戶端 RecvIV | 客戶端 | 解密從伺服器收到的封包 |
| 伺服器 SendIV | 伺服器 | 加密發送到客戶端的封包 |
| 伺服器 RecvIV | 伺服器 | 解密從客戶端收到的封包 |

### 第一個封包（Hello Packet）
- **永遠是明文**（未加密）
- 用於客戶端和伺服器交換 IV
- 包含版本號和 IV 值

### AES Handshake 流程
```
1. C→S: Hello Packet (plaintext)
   - 客戶端告知版本、SendIV、RecvIV

2. S→C: Hello Packet (plaintext)
   - 伺服器確認版本、回應 IV

3. 之後所有封包：MapleCrypto + Shanda + AES-OFB 加密
```

### v83 各版本加密變化

| 版本 | 加密方式 | 備註 |
|------|---------|------|
| v83~v117 | Shanda + AES-OFB | ✅ 仍在使用 |
| v118+ | 僅 AES-OFB | Shanda 已移除 |

### CMS (中國伺服器) v83 AES Key 參考
```java
// AES UserKey (first column per row)
0x13, 0x08, 0x06, 0xB4, 0x1B, 0x0F, 0x33, 0x52

// IV Transformation Key
0xC65053F2

// 客戶端版本
0x23 (35 decimal)
```

### 🆕 Hello Packet 握手完整流程 (botnored + MapleRef)

**URL**: https://mapleref.fandom.com/wiki/Packets
**URL**: https://botnored.weebly.com/blog/maplestory-classes-v83

> "The first packet sent / received will always be in plain-text (not encrypted)."

**握手流程圖**：
```
Client                              Server
  |                                    |
  |--- [1] Hello (明文) ------------->|
  |    - 版本號 (int)                 |
  |    - 發送 IV (byte[4])            |
  |    - 發送 AES Key (byte[16])     |
  |                                    |
  |<-- [2] ServerHello (明文) --------|
  |    - 版本確認                     |
  |    - 接收 IV (byte[4])           |
  |    - 接收 AES Key (byte[16])     |
  |                                    |
  |============ 加密通訊 =============|
```

**封包標頭格式** (v83)：
```
[4 bytes: Header][2 bytes: Opcode][Data...]

Header 格式：
[2 bytes: 封包長度][2 bytes: 混淆後的 Opcode]

Length 計算：
- 封包總長度 = Header(4) + Opcode(2) + Data length
- length ^= rawSeq (用於簡單混淆)
```

### 🆕 Wireshark 進階過濾技巧

**基本過濾**：
```

```

**進階過濾**：
```
# 只看從客戶端發出的封包
tcp.port == 8484 && ip.src == 127.0.0.1

# 只看 login 握手封包
tcp.port == 7575 && tcp.len > 0

# Follow TCP Stream (追蹤完整對話)
# 右鍵點擊封包 → Follow → TCP Stream

# 排除重複 ACK
tcp.analysis.duplicate_ack == 0

# 只看明文 Hello Packet (第一個封包)
tcp.payload[0:1] == 53  # 版本號 0x53 = 83
```
