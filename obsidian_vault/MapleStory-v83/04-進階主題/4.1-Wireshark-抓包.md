# 4.1 Wireshark 抓包教學

## 概述
使用 Wireshark 抓取 MapleStory 網路封包，分析客戶端與伺服器之間的通訊。

## Wireshark 下載
官網：https://www.wireshark.org/

## 抓包步驟

### 1. 設置過濾器
MapleStory 使用的連接埠通常為：
- 登入端口：7575
- 遊戲端口：8484 (或自定義)

### 2. 過濾表達式
```
tcp.port == 7575 || tcp.port == 8484
```

### 3. 分析重點
- **封包方向**：客戶端發送 (C→S) vs 伺服器發送 (S→C)
- **封包大小**：識別不同類型的操作
- **封包內容**：尋找可讀字串（Hello Packet 為純文字）

## 🐾 v83 封包加密架構

### 三層加密結構
```
┌─────────────────────────────────────────────────────┐
│  1. MapleCrypto (頭部混淆)                          │
│     - 防止靜態分析 opcode                           │
│     - Header: 4 bytes (長度 + 混淆 opcode)           │
│                                                        │
│  2. Shanda Encryption (對稱加密)                     │
│     - 早期版本使用                                   │
│     - 已於 v118+ 移除                               │
│                                                        │
│  3. AES-OFB (主加密)                                │
│     - v83 主要加密方式                              │
│     - 256-bit key, 4-byte IV                       │
└─────────────────────────────────────────────────────┘
```

### 重要原則
> **第一個發送/接收的封包永遠是純文字（未加密）**

這個未加密的封包就是 **Hello Packet**，用於協商加密參數。

## 📨 Hello Packet 詳細結構

### 客戶端 → 伺服器 (C→S) Hello
```
偏移    長度   內容
0x00    2      封包總長度 (little endian)
0x02    2      Maple 版本號 (如 0x0053 = 83)
0x04    n      版本字串 (如 "83")
n+4     4      SendIV (客戶端指定的發送 IV)
n+8     4      RecvIV (客戶端指定的接收 IV)
n+12    2      地區標誌 (MAPLE_LOCALE)
n+14    2      Unknown (未知用途)
```

### 伺服器 → 客戶端 (S→C) Hello
```
偏移    長度   內容
0x00    2      封包總長度
0x02    2      Maple 版本號
0x04    n      版本字串
n+4     4      RecvIV (伺服器回應)
n+8     4      SendIV (伺服器回應)
n+12    2      地區標誌
n+14    2      Unknown
```

### MapleCrypto 頭部混淆演算法
```python
# 加密端：對 opcode 進行混淆
def maple_encrypt_header(opcode, iv):
    verify = (opcode ^ iv[0]) + (iv[1] << 8)
    verify = (verify << 3) | (verify >> 5)  # 位旋轉 3 位
    return verify & 0xFFFF

# 解密端：反向操作
def maple_decrypt_header(encrypted, iv):
    verify = encrypted
    verify = (verify >> 3) | ((verify << 5) & 0xFFFF)  # 反向旋轉
    return (verify - (iv[1] << 8)) ^ iv[0]
```

## 📡 登入流程封包序列

### 完整登入流程
```
1. C→S: Hello Packet (純文字)
   → 客戶端告知版本、IV

2. S→C: Hello Packet (純文字)
   → 伺服器確認版本、回應 IV

3. C→S: 登入請求 (已加密)
   Opcode: 0x01 (LOGIN_PASSWORD)
   Data: 帳號 + 密碼 (已 AES 加密)

4. S→C: 登入結果 (已加密)
   Opcode: 0x00 (LOGIN_STATUS)
   Data: 結果碼 + 帳號資訊

5. C→S: 世界選擇 (已加密)
   Opcode: 0x05 (SELECT_CHARLIST)

6. S→C: 角色列表 (已加密)
   Opcode: 0x03 (CHARLIST)
   Data: 角色資料陣列

7. C→S: 選擇角色進入遊戲 (已加密)
   Opcode: 0x0B (PLAYER_DROP)
```

## 🔧 Wireshark 實戰技巧

### 設定顏色規則
```
紅色 (異常):    tcp.len > 2000      (大型封包)
綠色 (客戶端):  ip.src == 127.0.0.1 (本地發送)
藍色 (伺服器):  ip.dst == 127.0.0.1 (本地接收)
黃色 (登入):    tcp.port == 7575    (登入端口)
```

### Follow TCP Stream 分析
1. 右鍵任意封包 → Follow → TCP Stream
2. 檢視完整會話的明文部分（Hello Packet）
3. 分離 C→S 和 S→C 流量
4. 識別加密封包的長度規律

### 識別加密封包
```
徵兆：
- 封包長度固定 (如每次都是 48 bytes)
- 內容看起來像隨機位元組
- 相同操作產生的封包每次都不同 (因為 IV 變化)
```

## 🆕 完整登入流程封包序列 (v83 實戰)

**研究時間**：2026-03-24

### 完整 10 步登入封包序列
```
1. C→S Hello Packet (plaintext - 無 opcode header)
   - 告知版本、SendIV、RecvIV
   - 為純文字，Wireshark 直接可見

2. S→C Hello Packet (plaintext)
   - 伺服器確認版本、回應 IV

3. C→S LOGIN_PASSWORD (encrypted)
   Opcode: 0x01
   Data: 帳號字串 + 密碼字串 (AES-OFB 加密)

4. S→C LOGIN_STATUS (encrypted)
   Opcode: 0x00
   Data: 結果碼 + 帳號資訊

5. C→S SERVERLIST_REQUEST (encrypted)
   Opcode: 0x06
   - 請求世界伺服器列表

6. S→C SERVERLIST (encrypted)
   Opcode: 0x0C
   Data: 世界伺服器名稱、負載、人數上限

7. C→S CHARLIST_REQUEST (encrypted)
   Opcode: 0x07
   - 請求指定世界的角色列表

8. S→C CHARLIST (encrypted)
   Opcode: 0x03
   Data: 角色資料陣列（名稱、等級、職業、外觀）

9. C→S CHAR_SELECT (encrypted)
   Opcode: 0x08
   - 選擇角色

10. C→S PLAYER_DROP (encrypted)
    Opcode: 0x0B
    - 進入遊戲世界
```

### v83 登入結果碼（Login Status Code）

| 結果碼 | 意義 |
|--------|------|
| 0 | 登入成功 |
| 5 | 帳號或密碼錯誤 |
| 7 | 帳號已停用 |
| 13 | 刪除中（冷靜期） |
| 23 | 登入太頻繁（防機器人延遲） |
| 其他 | 未知錯誤 |

> 來源：RaGEZONE + 實戰分析

### 為什麼第一個封包是純文字？
- MapleStory 使用「加密握手」流程
- 第一個封包用於客戶端和伺服器**協商 IV（初始化向量）**
- 雙方需要用相同的 IV 初始化 AES-OFB 加密器
- 若加密，雙方都無法解密對方的第一個訊息

### Wireshark 分析技巧
- 使用 `tcp.port == 7575 || tcp.port == 8484` 過濾
- 在 Hello Packet 中確認版本號（如 0x0053 = 83）
- Follow TCP Stream 查看完整明文握手
- 加密封包長度固定（可用於識別類型）

## 🆕 CClientSocket::ProcessPacket 封包流向（RaGEZONE, 2019）

**來源**: https://forum.ragezone.com/threads/how-does-one-find-a-structure-of-a-packet-via-ida.1162654/

**所有客戶端接收封包的入口點**：
```
CClientSocket::ProcessPacket()
    ↓
根據 MapleCrypto header 解密
    ↓
提取 opcode (2 bytes, little endian)
    ↓
switch(opcode) 分發
    ↓
CLogin::OnPacket()    // 登入畫面
    或
CField::OnPacket()    // 遊戲中
```

**為什麼知道 Opcode 就能找到處理函數**：
- `CClientSocket::ProcessPacket` 內有一個大 switch 表
- 每個 case 對應一個 opcode
- case 內直接調用具體的 OnPacket 處理函數
- 因此知道 opcode 就能在 IDA 中快速定位函數

**封包處理函數命名慣例**：
| 函數前綴 | 狀態 | 說明 |
|---------|------|------|
| `CLogin::On*` | 登入畫面 | 登入相關封包處理 |
| `CField::On*` | 遊戲中 | 遊戲內封包處理 |
| `CShop::On*` | 商場 | 商場相關封包處理 |

### MapleCrypto + Shanda + AES 三層解密流程圖
```
發送端 (C→S):
明文資料
  → AES-OFB 加密 (使用版本 Key + IV)
  → Shanda 加密 (對稱加密)
  → MapleCrypto 頭部混淆 (4-byte header)
  → 網路傳輸

接收端 (S→C):
網路接收
  → MapleCrypto 解混淆 (提取 opcode + length)
  → Shanda 解密
  → AES-OFB 解密
  → 明文資料
```

### 排查解密失敗的檢查清單
| 檢查項目 | 正確值 | 症狀 |
|---------|--------|------|
| AES Key | 版本決定的 32-byte key | 解密出乱码 |
| IV | Hello Packet 協商的 IV | opcode 不匹配 |
| Shanda Key | 版本特定的對稱金鑰 | 資料順序錯誤 |
| 版本號 | 0x0053 (83) | Hello Packet 失敗 |
| Opcode 表 | 需與客戶端匹配 | switch case 走错分支 |
