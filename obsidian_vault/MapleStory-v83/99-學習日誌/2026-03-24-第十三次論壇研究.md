## 研究重點一：登入器（Login Redirector）原理更新

### RaGEZONE 2020 登入器教學（Pixeltris）

**來源**: https://forum.ragezone.com/threads/how-to-make-a-localhost-redirector.1192131/

**兩種主要方案**：

| 方案 | 原理 | 適用場景 |
|------|------|---------|
| **直接連線（非 Loopback）** | HxD 直接修改 MapleStory.exe 中的 IP 字串 | 簡單快速，IP 長度相同時 |
| **DLL 注入** | 注入 DLL 並覆寫記憶體中的 IP | 需要 Hook WSAConnect 等函數 |

**DLL 注入法原理**：
```
MapleStory.exe 載入
    ↓
LoadLibrary("mydll.dll")  ← 注入
    ↓
mydll.dll 中 Hook WSAConnect()
    ↓
當客戶端嘗試連線時，Hook 函數攔截並修改目標 IP/Port
    ↓
redirect 到 127.0.0.1:7575
```

**Pixeltris 的兩個 IP 解決方案**：
1. **127.0.0.1 專用方案**：不需要完整 DLL 注入，僅需 Hook `connect()` 函數並檢查目標是否為 127.0.0.1
2. **通用 DLL 注入方案**：可覆寫記憶體中的任意 IP，支援遠程連線

## 研究重點二：Opcode 結構更新

### CClientSocket::ProcessPacket 封包分發流程

**來源**: RaGEZONE - How to find packet structures via IDA

**核心概念**：所有客戶端接收的封包都經過同一個入口函數 `CClientSocket::ProcessPacket`

**封包流向**：
```
網路接收 (TCP)
    ↓
CClientSocket::ProcessPacket()
    ↓  根據 MapleCrypto header 解密
    ↓  提取 opcode (2 bytes)
    ↓
switch(opcode)  // CLogin 或 CField 根據狀態分發
    ↓
CLogin::OnPacket()      // 登入畫面狀態
    或
CField::OnPacket()      // 遊戲中狀態
```

**這就是為什麼有 opcode 清單就能輕鬆找到客戶端接收封包的處理函數**

### CInPacket 解碼流程

**來源**: RaGEZONE - Java flow of MapleStory functions

**封包解碼流程**：
```
1. CClientSocket 接收 TCP 資料
2. CInPacket::Decode() 被調用
   - 解析 MapleCrypto header (4 bytes)
   - 解密並提取 opcode (2 bytes)
   - 剩餘為封包內容 (payload)
3. CInPacket 構造完成 (nType = opcode, buffer = payload)
4. 根據 nType 分發到具體處理函數
5. 在處理函數中讀取 buffer 的各個欄位
```

**IDA 分析技巧**：
- 在 IDA 中搜尋 `CInPacket::` 相關函數
- `Decode1` → 讀取 1 byte
- `Decode2` → 讀取 2 bytes (short/opcode)
- `Decode4` → 讀取 4 bytes (int)
- `DecodeBuffer(x, n)` → 讀取 n bytes

### MapleRef Wiki - Opcode 結構說明

**來源**: https://mapleref.fandom.com/wiki/Operation_Code

**重點摘要**：
- Opcode 是封包開頭用於識別類型的唯一識別符
- v83 使用 **2 bytes (short)** 而非早期版本的 1 byte
- 這將最大 Opcode 數量從 255 (2^8) 擴展到 65,536 (2^16)
- 不同版本有不同的 Opcode 集合，需要匹配

### v83 更新 Opcodes 討論（RaGEZONE 2016）

**來源**: https://forum.ragezone.com/threads/updating-opcodes.1112067/

**重要發現**：
- `CLogin::OnPacket` 是所有登入相關客戶端接收封包的基礎處理類
- `CInPacket` 的第三個參數是 `iPacket`，指向封包本身
- 正確理解 IDA 中的 local types 和 PDB 可大幅加速分析

### WZ 編輯常見問題：Import 新版物品到 v83

**來源**: RaGEZONE 2025-11

**問題**：新版物品 Import 到 v83 時，物品可以生成但無法裝備

**原因分析**：
1. 物品的 `.img` 檔案 Import 時結構不完整
2. Character.wz 中缺少對應的裝備定義
3. 物品依賴的新屬性在 v83 客戶端中不存在（如某些 v100+ 的裝備屬性）
4. `lvlm`、`STR`、`DEX` 等限制屬性設置不正確

**解決方案**：
- 確保導入完整的 .img 文件及其所有依賴
- 檢查物品的 `lvlm`、`STR`、`DEX` 等屬性
- 確認 WZ 結構版本匹配
- 盡量從同版本 WZ 提取而非跨版本

### Quest 等級限制移除詳細步驟

**來源**: RaGEZONE 2025-07

**詳細步驟**：
```
1. 用 Harepacker 開啟 Quest.wz，encryption: GMS (old)
2. 導航到要修改的任務目錄
3. Tools → Export selection → XML → Classic
   （重要：選擇 Classic，否則會使用新版加密）
4. 用 Notepad++ 開啟 XML，搜尋所有 lvmax 屬性並移除
5. 存檔
6. 用 Harepacker 重新導入
```

## 研究重點四：NPC 腳本源相容性問題

### 不同音源（Cosmic/Kinoko/HeavenMS）腳本相容性

**來源**: RaGEZONE 2026-03

**問題**：
- 不同音源的 NPC 腳本解析器版本不同
- 較舊的腳本在新解析器可能不相容
- 原因：腳本解析器實現差異

**解決方案**：
1. 在 source 中 include 較舊版本的 script reader
2. 使用相容格式編寫腳本（避免特定版本專有函數）
3. 先在本地測試再部署

**最佳實踐**：
- 使用普遍支援的 `cm.` 函數
- 避免使用特定音源專有的 Package 路徑
- 避免 `importPackage`（JDK 7+ 已廢棄）
