## 🆕 學習重點一：Ezorsia v2 DLL 注入方案（2026 最新）

### Ezorsia v2 - 史上第一個獨立 DLL 客戶端

**GitHub**: https://github.com/444Ro666/MapleEzorsia-v2
**官方 Wiki**: https://github.com/444Ro666/MapleEzorsia-v2/wiki/v83%E2%80%90Client%E2%80%90Setup%E2%80%90and%E2%80%90Development%E2%80%90Guide

**突破性特點**：
> "this is the first ever standalone dll client ever released (at least for v83)"
> — RaGEZONE, 2023-11

這是 v83 歷史上第一個真正意義上的獨立 DLL 客戶端，無需修改原始 exe，透過 `dinput8.dll` hook 實現。

### Ezorsia v2 工作原理

```
┌─────────────────────────────────────────────┐
│  Ezorsia v2 DLL (dinput8.dll)              │
│                                             │
│  1. Hook dinput8.dll (第二早加載的 DLL)     │
│  2. 修改記憶體中的客戶端位址                 │
│  3. 啟用 HD 解析度                          │
│  4. 自動載入 Ezorsia WZ edits               │
│  5. 自動連接到設定的伺服器 IP               │
└─────────────────────────────────────────────┘
```

### 與傳統 localhost.exe 的比較

| 項目 | 傳統 localhost.exe | Ezorsia v2 DLL |
|------|-------------------|-----------------|
| 修改方式 | 直接修改/替換 exe | DLL 注入 |
| 安全性 | 較低（可能被標記病毒） | 較高（dll hook 方式） |
| HD 支援 | 需額外設定 | 內建 |
| WZ 衝突 | 常見 | 自動處理 |
| 源代碼 | 通常不開源 | AGPL 開源 |
| IP 修改 | 需 Hex 編輯 | 自動處理 |
| 維護性 | 困難 | 簡單 |

### Ezorsia v2 完整架設流程 (2025-2026 最新)

**前置準備**：
```
1. 安裝預設 v83 客戶端（不要刪除任何檔案）
2. 下載 Ezorsia v2 DLL（從 releases 頁面）
3. 下載 Hendi's 4G 客戶端（啟用 4GB 記憶體）
```

**Hendi 4G 客戶端 Hex 編輯（可選）**：
```
位置: 0000013E
原始值: 0f
修改為: 2f

目的: 啟用 "Large Address Space Aware" (LAA) 旗標
效果: 客戶端可用到 4GB 記憶體，而非僅 2GB
```

**admin elevation 移除（可選）**：
- 在 ezorsia config 中設定，或手動 Hex 編輯
- 若自行編輯，之後需將 exe 改名（如 MapleStory2.exe）

**config.ini 設定**：
```ini
[Connection]
serverIP=127.0.0.1
serverPort=8484

[Display]
width=1280

[Advanced]

```

### Ezorsia v2 安全警告（重要）

**來自開發者的重要安全說明**：

1. **不要下載無源代碼的第三方 Ezorsia 版本**
   - DLL 可被用於惡意目的（鍵盤記錄、比特幣挖礦等）
   - 務必從官方 releases 或自行編譯

2. **不要使用 Themida/VMProtect 打包的 DLL**
   - 可能隱藏惡意程式碼
   - 難以在調試環境中檢測

3. **推薦使用 Magpie 實現無邊框全螢幕**
   - 下載：https://github.com/Blinue/Magpie
   - 在遊戲運行時開啟，設定 auto scale when in foreground
   - 避免 UI 元素縮放問題

### Ezorsia v2 WZ 處理機制

**從 2023/11/21 起**：
- Ezorsia V2 使用自訂 WZ/IMG，不會與任何伺服器的檔案衝突
- 無需手動合併 WZ edits

**早期版本需手動合併 UI.wz**：
```
1. 用 HaRepacker 同時打開 EzorsiaV2_UI.wz 和你的 UI.wz
2. 選擇 MapleEzorsiaV2wzfiles.img
3. 複製粘貼到你的 UI.wz
4. 若使用 IMG 格式，導出為 IMG 並放入 Data 資料夾
```

## 🆕 學習重點二：DLL Proxy 注入原理

### 代理 DLL 注入技術（RaGEZONE, 2016）

**來源**: https://forum.ragezone.com/threads/injecting-code-inside-maplestory-using-a-dll-proxy.1111732/

**原理**：
MapleStory 依賴多個 Windows DLL（如 dinput9.dll、kernel32.dll 等），透過代理 DLL 可以：

```
┌──────────────────────────────────────────┐
│  正常載入鏈:                             │
│  MapleStory.exe → dinput9.dll (真實)     │
│                                          │
│  代理載入鏈:                             │
│  MapleStory.exe → dinput9.dll (我的)     │
│       ↓                                  │
│  我的 dinput9.dll 再調用真實的 dinput9   │
└──────────────────────────────────────────┘
```

**優點**：
- 不需要傳統的 DLL injection 工具
- 不會被記憶體掃描檢測到
- 可在遊戲啟動時自動載入

**實作方式**：
```cpp
// 我的 dinput9.cpp (代理 DLL)
// 匯出所有原始 dinput9.dll 函數
extern "C" {
    __declspec(dllexport) HRESULT __stdcall DirectInput8Create(
        HINSTANCE hinst, DWORD dwVersion, REFIID riidltf,
        LPVOID *ppvOut, LPUNKNOWN punkOuter
    ) {
        // 在這裡添加自訂邏輯
        // 然後呼叫真實的 dinput9.dll
        return Real_DirectInput8Create(...);
    }
    // ... 其他所有 dinput9 函數同理
}
```

### DLL 代理與 Ezorsia v2 的區別

| 技術 | 原理 | 適用場景 |
|------|------|---------|
| DLL Proxy | 替換系統 DLL（如 dinput9.dll）| 簡單修改 |
| Ezorsia v2 | Hook dinput8.dll 修改記憶體 | HD + 複雜修改 |
| Themida 殼 | 混淆 + VMProtect | 保護原創程式 |

### HxD Hex 編輯 localhost.exe 詳細步驟

**使用 HxD 的正確流程**：
```
1. Ctrl+F → 搜尋類型選擇 "Text-string"
2. 搜尋 "127.0.0.1"
3. 會找到三個連續出現的 "127.0.0.1"
4. 將三處都替換為目標 IP
5. 存檔
```

**⚠️ IP 長度一致性原則**：
- `127.0.0.1` = 9 bytes
- 若目標 IP 長度不同，必須用 `\x00` 填充
- 或使用 STREDIT 自動處理長度問題

### STREDIT（推薦工具）

- 自動處理 IP 長度不一致問題

## 🆕 學習重點五：Wireshark 抓包進階技巧

### 識別加密封包的技巧

**為什麼相同操作產生的封包每次長度都不同？**
```
原因：IV (初始化向量) 每次都會改變
每次加密時，IV 都會遞增/更新
導致相同 opcode 的封包看起來長度不同

但實際上：
- Opcode 固定 (2 bytes)
- Payload 內容長度固定
- 只有 MapleCrypto header 的干擾位元組不同
```

### 封包識別特徵表

| 徵兆 | 封包類型 |
|------|---------|
| 長度固定 (如每次 32 bytes) | 可能是 MOVE_PLAYER (0x14) |
| 包含可讀字串 | 可能是 CHAT (聊天) |
| 第一個 byte 為 0x00 | 伺服器回應封包 |
| 封包包含 00 00 00 00 | 可能是 PING/心跳封包 |
| 固定長度 + 00 byte | 可能是 MOB_MOVE |

### Wireshark 進階過濾表達式

```wireshark
// 基本過濾
tcp.port == 8484

// 只看客戶端發送到伺服器
tcp.srcport == xxxx && tcp.dstport == 8484

// 只看大封包（可能是地圖載入）

// 組合過濾

Right-click → Follow → TCP Stream
```

## 🆕 學習重點六：v83 Opcode 結構深入分析

### MapleCrypto Header 加密/解密公式

**加密端（伺服器/客戶端發送）**:
```java
public byte[] getPacketHeader(int opcode, int length, int iv) {
    int verify = (opcode ^ (iv & 0xFF)) + ((iv >> 8) & 0xFF) * 0x100;
    verify = (verify << 3) | (verify >>> 5);
    verify &= 0xFFFF;
    return new byte[] {
        (byte) (verify & 0xFF),
        (byte) ((verify >>> 8) & 0xFF),
        (byte) (length & 0xFF),
        (byte) ((length >>> 8) & 0xFF)
    };
}
```

**解密端（接收方）**:
```java
public int retrieveOpcode(byte[] header, int iv) {
    int verify = (header[0] & 0xFF) | ((header[1] << 8) & 0xFF00);
    verify = (verify >>> 3) | ((verify << 5) & 0xFFFF);
    return (verify - ((iv >> 8) & 0xFF) * 0x100) ^ (iv & 0xFF);
}
```

### CInPacket 封包接收完整流程

```cpp
// 客戶端接收封包 (IDA pseudocode)
void __thiscall CClientSocket::ProcessPacket(CClientSocket *this) {
    while (true) {
        // 讀取 4-byte header
        BYTE header[4];
        recv(socket, header, 4);
        
        // MapleCrypto 解混淆
        int packetLen = MapleCrypto::decryptLength(header, ivRecv);
        int opcode = MapleCrypto::decryptOpcode(header, ivRecv);
        
        // 分配 buffer
        byte *buffer = new byte[packetLen];
        
        // 接收剩餘資料 (packetLen bytes)
        recv(socket, buffer, packetLen);
        
        // AES-OFB + Shanda 解密
        MapleCrypto::decrypt(buffer, packetLen, ivRecv);
        
        // 構造 CInPacket
        CInPacket packet;
        packet.nType = opcode;      // 設定 opcode
        packet.buffer = buffer;      // 設定 payload
        
        // 分發到處理函數
        switch (opcode) {
            case 0x00: CLogin::OnPacket(&packet); break;
            case 0x01: CLogin::OnLoginPassword(&packet); break;
            // ...
        }
    }
}
```

### 動態 NPC 商店腳本（不需 SQL）

**來源**: https://forum.ragezone.com/threads/dynamic-npc-shop-script-v83.1089979/

```javascript
/*
 * 動態 NPC 商店 - 不需要 SQL
 * NPC ID: 9010009
 */
var status = 0;
var items = Array(
    Array(1302000, 10000, 1),   // [物品ID, 價格, 庫存]
    Array(1302001, 20000, 1),
    Array(1302002, 30000, 1),
    Array(1302003, 50000, 1)
);

function start() {
    status = -1;
    action(1, 0, 0);
}

function action(mode, type, selection) {
    if (mode == -1) {
        cm.dispose();
    } else {
        if (mode == 0 && type == 1) {
            cm.dispose();
            return;
        }
        
        if (mode == 1)
            status++;
        else
            status--;
        
        if (status == 0) {
            var msg = "歡迎光臨！\r\n請選擇商品：\r\n";
            for (var i = 0; i < items.length; i++) {
                msg += "#L" + i + "##i" + items[i][0] + "# - " + items[i][1] + " 金幣#l\r\n";
            }
            cm.sendSimple(msg);
        } else if (status == 1) {
            var item = items[selection];
            if (cm.getMeso() >= item[1]) {
                if (cm.canHold(item[0])) {
                    cm.gainMeso(-item[1]);
                    cm.gainItem(item[0], item[2]);
                    cm.sendOk("購買成功！");
                } else {
                    cm.sendOk("背包空間不足！");
                }
            } else {
                cm.sendOk("金幣不足！");
            }
            cm.dispose();
        }
    }
}
```
