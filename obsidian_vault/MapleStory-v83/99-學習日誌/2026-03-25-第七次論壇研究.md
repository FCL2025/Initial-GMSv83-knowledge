**可能原因**：
- 防火牆未開放 8484 端口
- 客戶端 IP 指向錯誤（需指向 PC1 的 LAN IP 而非 127.0.0.1）
- MySQL 資料庫未授權遠端連線

**解決方案**：
```sql
-- MySQL 授權遠端存取
GRANT ALL PRIVILEGES ON *.* TO 'root'@'%' IDENTIFIED BY 'password';
FLUSH PRIVILEGES;
```

```batch
-- Windows 防火牆開放端口
netsh advfirewall firewall add rule name="MapleStory" dir=in action=allow protocol=tcp localport=8484
```

### 3) Opcode結構

#### 🆕 v83 封包包長度識別技巧 (2026 研究)
**關鍵問題**：為什麼相同操作產生的封包每次長度都不同？

**原因**：IV (初始化向量) 每次都會改變
```
每次加密時，IV 都會遞增/更新
導致相同 opcode 的封包看起來長度不同

但實際上：
- Opcode 固定 (2 bytes)
- Payload 內容長度固定
- 只有 MapleCrypto header 的干擾位元組不同
```

**識別封包類型的技巧**：
| 徵兆 | 封包類型 |
|------|---------|
| 長度固定 (如每次 32 bytes) | 可能是 MOVE_PLAYER (0x14) |
| 包含可讀字串 | 可能是 CHAT (聊天) |
| 第一個 byte 為 0x00 | 伺服器回應封包 |
| 封包包含 00 00 00 00 | 可能是 PING/心跳封包 |

**Wireshark 過濾技巧**：
```
// 只看 MapleStory 相關流量
tcp.port == 7575 || tcp.port == 8484

// 只看客戶端發送
tcp.srcport == xxxx && tcp.dstport == 7575

// Follow TCP Stream
Right-click → Follow → TCP Stream
```

#### 🆕 CMS 中國伺服器 v83 登入封包解密實戰 (RaGEZONE, 2018)
**來源**: https://forum.ragezone.com/threads/help-remove-encrypt-decrypt-in-client-cms-chinese-maplestory.1155998/

**三層加密流程**：
```
客戶端發送原始密文 (C→S)
  → AES-OFB 解密
  → Shanda Decryption
  → 明文 opcode + data
```

**實際資料**：
```
原始密文: 64 A4 9F 07 FD 4D 6C 49...
AES解密後: 71 1A 5B 6A 7E B9 C9 EA...
Shanda解密後: E2 FB 40 7E 67 08 B6 77...
→ opcode = 0xFBE2 (預期 0x0001)
→ 版本不匹配！
```

**CMS v83 AES Key 參考**：
```java
// AES UserKey
0x13, 0x08, 0x06, 0xB4, 0x1B, 0x0F, 0x33, 0x52

// IV Transformation Key
0xC65053F2

// 客戶端版本
0x23 (35 decimal)
```

**排查解密失敗的檢查清單**：
| 檢查項目 | 正確值 | 症狀 |
|---------|--------|------|
| AES Key | 版本決定的 32-byte key | 解密出乱码 |
| IV | Hello Packet 協商的 IV | opcode 不匹配 |
| Shanda Key | 版本特定的對稱金鑰 | 資料順序錯誤 |
| 版本號 | 0x0053 (83) | Hello Packet 失敗 |

#### 🆕 跨版本物品 Import 問題 (RaGEZONE, 2025-11)
**來源**: https://forum.ragezone.com/threads/maplestory-v83-newer-items-import.1256083/

**問題**：將 v100+ 物品 Import 到 v83 時，物品可以生成但無法裝備

**根本原因**：
| 原因 | 說明 |
|------|------|
| .img 結構不完整 | 較新版本包含 v83 不支援的屬性節點 |
| Character.wz 缺少定義 | 裝備需要對應的造型定義 |
| 新屬性不存在 | v83 客戶端無法識別某些 v100+ 屬性 |
| 屬性值不合規範 | lvlm、STR、DEX 等有不同範圍限制 |

**正確流程**：
```
1. 從同版本（v83）WZ 提取物品
2. 如果必須跨版本：
   - 提取完整的 .img 文件及其所有依賴
   - 移除 v83 不支援的屬性節點
   - 確認 lvlm、STR、DEX 在合理範圍內
3. 在 Character.wz 中新增對應的裝備造型定義
4. 在 String.wz 中新增物品名稱
```

**v83 裝備必備屬性檢查清單**：
- [ ] `info/lvlm` - 等級上限
- [ ] `info/STR` - 力量需求
- [ ] `info/DEX` - 敏捷需求
- [ ] `info/INT` - 智力需求
- [ ] `info/LUK` - 幸運需求
- [ ] `info/tuc` - 可強化次數
- [ ] `info/cursed` - 詛咒標誌
- [ ] `info/only` - 職業限制

### 5) NPC腳本編寫

#### 🆕 Kinoko NPC Scripts 相容性討論 (RaGEZONE, 2026-03)
**來源**: https://forum.ragezone.com/threads/kinoko-npc-scripts.1261538/

**問題分析**：
- 不同音源（Cosmic、Kinoko、HeavenMS）的 NPC 腳本解析器版本存在差異
- 較舊的腳本在新解析器可能不相容

**常見不相容程式碼**：
```javascript
// ❌ 不相容 (JDK 7+)
importPackage(Packages.client);
importPackage(Packages.server);

// ✅ 相容寫法
var MapleCharacter = Packages.client.MapleCharacter;
```

**解決方案**：
1. 使用普遍支援的 `cm.` 函數
2. 避免使用特定音源專有的 Package 路徑
3. 移除 `importPackage`（JDK 7+ 已廢棄）
4. 先在本地測試再部署到正式伺服器

#### 🆕 快速傳送 NPC 腳本源 (RaGEZONE, 2020)

var status = 0;

function start() {

function action(mode, type, selection) {

        cm.warp(220000000);  // 傳送到 Ludi

    }
}
