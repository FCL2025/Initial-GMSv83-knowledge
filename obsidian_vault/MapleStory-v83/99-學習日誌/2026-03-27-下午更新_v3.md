## 二、登入器/客戶端修改

### 2.1 v83 AES 加密機制深度解析

**來源**：
- GitHub kkartaltepe/chatty-v83 `MapleAESOFB.java`
- GitHub razfriman/SnowMS `MapleAESOFB.java`
- RaGEZONE 討論串

#### AES OFB 模式加密原理

MapleStory v83 使用 **AES OFB (Output Feedback)** 加密模式，核心特性：

```
MapleAESOFB 構造函數：
- key[]: 256-bit AES 金鑰（32 bytes）
- iv[]: 4-byte 初始化向量
- mapleVersion: 版本號（v83 = 83）

版本號位元組反轉：
this.mapleVersion = (short) (((mapleVersion >> 8) & 0xFF) | ((mapleVersion << 8) & 0xFF00));
// 83 (0x53) → 0x5300 類似的處理
```

#### 登入握手序列

根據 maplelib (Go) 文档：

> MapleStory uses two keys: **one is used to encrypt sent packets and the other is used to decrypt received packets**. The two keys are randomly generated when a client connects and sent in the unencrypted handshake packet which is the first packet.

**完整登入流程**：

```
客戶端                        伺服器
   |                             |
   |--- [Hello Packet (明文)] -->|
   |    版本號 + 隨機IV           |
   |                             |
   |<-- [Challenge/接受] --------|
   |    伺服器IV + 会话ID         |
   |                             |
   |=== [加密通道建立] ===========|
   |    後續所有封包皆加密        |
```

#### 登入封包結構 (Hello Packet)

根據 RaGEZONE 討論（v83 OdinMS）：

```
Hello Packet 範例（十六進制）：
0E 00 23 00 01 00 31 46 72 7A 52 52 30 78 73 04

分析：
- 0E 00        → length (little-endian) = 14 bytes
- 23 00        → version = 0x0023 = 35 (此為改過的版本號)
- 01 00        → SUB_VERSION 或 類型
- 31 46 72 7A  → IV (recv) = "1FrzR"
- 52 30 78 73  → IV (send) = "R0xs"
- 04           → MapleType 或 補丁標記

注意：v83 客戶端的 MAPLE_VERSION = 83 (0x53)
```

public class MapleAESOFB {
    private byte iv[];
    private Cipher cipher;
    private short mapleVersion;

## 三、抓包分析

### 3.1 Maple_Pshark — v83 專用 Packet Logger

**GitHub**: https://github.com/obstriker/Maple_Pshark

**功能**：
1. **攔截 Recv 封包** — 從伺服器接收的加密封包
2. **修改封包** — 即時編輯攔截到的封包內容
3. **注入封包** — 對 Send 或 Recv 注入自定義封包
4. **規則過濾** — 設定條件過濾特定封包

**原理**：
> This piece of software hooks to the encryption function and let you intercept messages as they go.

- Hook 到客戶端的加密函數
- 在封包加密/解密前攔截
- 適合 v83 MapleStory 私服環境

### 3.2 MapleShark — SharpPcap 封包嗅探器

**GitHub**: https://github.com/diamondo25/MapleShark

- 使用 SharpPcap (.NET) 進行網卡層抓包
- 不同於 Maple_Pshark（應用層 Hook）
- 可抓到完整的 TCP 流量

### 3.3 MaplePacketPuller — Python 分析工具

**GitHub**: https://github.com/Bratah123/MaplePacketPuller

- Python 實現的 MapleStory 封包結構分析器
- 自動解析封包格式

### 3.4 Wireshark 抓包技巧

**Snow Sniffer** (RaGEZONE, 2010)：

```
使用流程：
1. 設定 packets.properties 攔截規則
2. 啟動 Decrypt Packets.bat
3. 啟動 MapleStory.exe
4. log.txt 自動記錄封包

⚠️ 現代私服可能需要 SSL/TLS 攔截或特殊代理
```

**關鍵原則**：

> The first packet sent / received will always be in plain-text (not encrypted).

- 第一個封包（Hello/Handshake）為明文
- 之後所有封包使用 AES OFB 加密
- 理解加密起始點是分析封包的關鍵

## 四、Opcode 結構

### 4.1 Opcode 基本概念

根據 MapleRef Fandom：

> Actions that are handled by packets on MapleStory are defined by an identifier called an Operation Code (opcode for short), which is **the first 2 bytes of the packet (a short)**, though in earlier versions it used to be the first byte (a byte).

**v83 使用 2-byte Opcode**（版本差異）：
- 早期版本：1-byte Opcode
- v83 及之後：2-byte Opcode

### 4.2 封包結構

**Header 結構** (v83)：

```
┌─────────────────────────────────────┐
│ Header (4 bytes)                    │
├──────────┬──────────┬───────────────┤
│ IV加密段 │ 長度(2B) │ 版本校驗(2B)  │
└──────────┴──────────┴───────────────┘
│ Opcode (2 bytes)                    │
├─────────────────────────────────────┤
│ Payload (可變長度)                  │
└─────────────────────────────────────┘

根據 RaGEZONE 討論：
"Packet headers are 4 bytes long and are generated by the send IV, 
packet length without the header, and the version of MapleStory."
```

### 4.3 IDA 分析 Opcode

根據 RaGEZONE 教程：

> All client-received packets go through **CClientSocket::ProcessPacket**, and from there it looks at the header and goes to **CField::OnPacket, CLogin::OnPacket**, etc.

**分析步驟**：
1. 使用 IDA 找到 `CClientSocket::ProcessPacket`
2. 查看 packet handler 表
3. 識別 CLogin/CField 等模組的 OnPacket 函數
4. 交叉引用找到各 Opcode 的處理邏輯

### 4.4 常用 Opcode 範例 (v83)

根據 MapleRef Wiki 和討論：

| Opcode 名稱 | 值 (Hex) | 方向 | 說明 |
|------------|----------|------|------|
| LOGIN_REQUEST | 0x01 | C→S | 登入請求 |
| LOGIN_RESPONSE | 0x00 | S→C | 登入回應 |
| CHAR_SELECT | 0x04 | C→S | 角色選擇 |
| CHAR_LIST | 0x03 | S→C | 角色列表 |
| ENTER_PORTAL | 0x08 | C→S | 進入傳送門 |
| MOVE_PLAYER | 0x26 | C→S | 玩家移動 |
| SPAWN_NPC | 0x3C | S→C | 生成 NPC |
| NPC_TALK | 0x3A | S→C | NPC 對話 |

## 五、WZ 修改

### 5.1 WZ 文件結構

**v83 使用加密的 .wz 格式**：
- 舊版 GMS 使用特殊加密
- 需要 HaRepacker 4.2.2+ 處理
- 不同於現代 XML 格式

### 5.2 HaRepacker 使用要點

**來源**: GitHub jonnylin13/Maple83

```
使用流程：
1. HaRepacker 4.2.2 開啟 String.wz
2. 導航到目標路徑 (如 "Character/立即查看.wz/...")
3. 直接修改數值（避免刪除/重建節點！）
4. 儲存
5. 使用 "Private Server..." 導出選項
```

### 5.3 WZ 編輯風險提示

根據 v83 私服教程：

> DO NOT USE the server's XMLs for reimporting into the client's WZ, it WILL generate bugs.

**正確流程**：
```
客戶端 WZ 修改：
1. 原始加密 WZ → HaRepacker 直接編輯 → 存回加密格式

伺服器端：
1. 導出 XML → 覆蓋伺服器的 WZ.XML 目錄
2. 不要用伺服器 XML 倒灌回客戶端 WZ！
```

### 5.4 skill.wz 編輯注意事項

根據 Reddit r/mapleservers 討論 (2024)：

> You're going to have to use assembly. It's a much steeper learning curve than just editing .wz files. Ollydbg is a free x86 debugger.

**skill.wz 這類功能性 WZ**：
- 可能需要組合 ASM 修改
- Ollydbg 是常用調試工具
- 不只是簡單的數值修改

### 6.2 Quest NPC 腳本模式

**來源**: RaGEZONE "How to script Quest NPCs [v83]"

**關鍵概念**：
- 使用 `status` 變量追蹤對話狀態
- 每個 `send*` 方法遞增 status
- 根據 status 和 selection 處理分支

```javascript
// 範例結構
function start() {
    status = 0;
    cm.sendNext("歡迎來到任務！");
}

function action(mode, type, selection) {
    if (mode == -1) {
        cm.dispose();
        return;
    }
    
    if (mode == 0) {
        status--;
    } else {
        status++;
    }
    
    if (status == 0) {
        // 初始對話
    } else if (status == 1) {
        // 處理選擇
    } else if (status == 2) {
        // 完成任務
    }
}
```
