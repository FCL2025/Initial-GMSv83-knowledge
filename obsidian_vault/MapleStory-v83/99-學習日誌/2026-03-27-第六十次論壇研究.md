## 📌 2) 抓包分析

### MapleStory 封包三層加密架構 (v83)

```
┌──────────────────────────────────────┐
│  Layer 1: AES-OFB (主要加密)          │
│    - 256-bit key                     │
│    - IV 每次遞增                      │
├──────────────────────────────────────┤
│  Layer 2: Shanda 加密 (中國版)        │
│    - 對稱加密演算法                   │
│    - v83 仍在使用                     │
├──────────────────────────────────────┤
│  Layer 3: MapleCrypto 混淆           │
│    - Header 混淆                      │
│    - 版本號 + IV 關聯                 │
└──────────────────────────────────────┘
```

### 封包格式 (v83)
```
[Header (4 bytes)][Opcode (2 bytes)][Data (變長)]

Header 結構:
  - 2 bytes: 封包長度 (混淆後)
  - 2 bytes: Opcode (混淆後)
```

### 關鍵：Hello Packet 是明文
- 連接建立後第一個封包**未加密**
- 用於交換 AES 初始化向量 (IV)
- 客戶端和伺服器各自生成隨機密鑰

### MapleCrypto 關鍵函數 (IDA 函數對照)
| IDA 函數 | MaplePacketWriter 方法 | 讀取大小 |
|---------|----------------------|---------|
| `Decode1` | `write` | 1 byte |
| `Decode2` | `writeShort` | 2 bytes |
| `Decode4` | `writeInt` | 4 bytes |
| `DecodeStr` | `writeMapleAsciiString` | 可變 |
| `DecodeBuffer(x, 8)` | `writeLong` | 8 bytes |

### CInPacket 封包接收完整流程 (IDA 分析)
```cpp
void __thiscall CClientSocket::ProcessPacket(CClientSocket *this) {
    while (true) {
        BYTE header[4];
        recv(socket, header, 4);
        
        // MapleCrypto 解混淆
        int packetLen = MapleCrypto::decryptLength(header, ivRecv);
        int opcode = MapleCrypto::decryptOpcode(header, ivRecv);
        
        byte *buffer = new byte[packetLen];
        recv(socket, buffer, packetLen);
        
        // AES-OFB + Shanda 解密
        MapleCrypto::decrypt(buffer, packetLen, ivRecv);
        
        CInPacket packet;
        packet.nType = opcode;
        packet.buffer = buffer;
        
        switch (opcode) {
            case 0x00: CLogin::OnPacket(&packet); break;
            case 0x01: CLogin::OnLoginPassword(&packet); break;
        }
    }
}
```

## 📌 3) Opcode結構

### Opcode 處理函數 (IDA 分析)

| 函數 | 位址 (v83 Rev 3.00) | 說明 |
|------|-------------------|------|
| `CClientSocket::ProcessPacket` | - | 封包分發入口 |
| `CLogin::OnPacket` | 0x0051E062 | 登入相關 opcode |
| `CWvsContext::OnPacket` | 0x0075055B | 登入後到 warp2map |
| `CField::OnPacket` | 0x004D7212 | 遊戲中所有 opcode |
| `CInPacket::Decode1` | 0x0041ED97 | 讀 1 byte |
| `CInPacket::Decode2` | 0x0041EE0C | 讀 2 bytes |
| `CInPacket::Decode4` | 0x0041EE84 | 讀 4 bytes |
| `CInPacket::DecodeBuffer` | 0x0041EEFB | 讀緩衝區 |
| `CInPacket::DecodeStr` | 0x0044B100 | 讀字串 |

### v83 Login Opcode (CLogin::OnPacket Switch Cases)

| case 值 | Opcode (Hex) | 函數名稱 |
|---------|-------------|---------|
| 0 | 0x00 | OnCheckPasswordResult |
| 1 | 0x01 | (未知) |
| 2 | 0x02 | (未知) |
| ... | ... | ... |

### 所有客戶端接收封包都經過的函數鏈
```
CClientSocket::ProcessPacket
    ↓
根據 header 分發到:
    - CLogin::OnPacket (登入 opcode)
    - CWvsContext::OnPacket (warp2map 之前的 opcode)
    - CField::OnPacket (遊戲中所有其餘 opcode)
```

### 為什麼 CLogin::OnPacket 很重要？
- 所有**登入相關** opcode 都由此函數處理
- 找到這個函數等於找到登入 opcode 的 switch case
- 使用 IDA 交叉引用可以找到所有 handler

### IDA 分析技巧
1. **靜態分析**：找到 CLogin::OnPacket 的 switch case
2. **動態分析**：在 CInPacket::Decode2 設斷點，觀察 opcode 值
3. **交叉引用**：每個 case 內部會調用具體的 OnXXX 函數

## 📌 4) WZ修改

### 🆕 MapleSolaxiaV2 WZ 編輯黃金法則

> **DO NOT USE the server's XMLs for reimporting into the client's WZ, it WILL generate bugs.**

**原因**：
- 客戶端 WZ 使用 **GMS (old)** 加密格式
- 伺服器 WZ.XML 是**解密的明文**格式
- 兩者結構不同，直接覆蓋會導致客戶端崩潰

### WZ 編輯正確流程
```
1. HaRepacker 4.2.4 開啟 GMS (old) 加密的客戶端 WZ
2. 使用 UI 直接修改數值（禁止刪除/重新導入節點！）
3. 儲存 WZ
4. 在 HaRepacker 中選擇 "Private Server..." 導出為 XML
5. 將導出的 XML 覆蓋到伺服器的 WZ.XML 目錄
```

### 錯誤流程（不要這樣做！）
```
❌ 直接用伺服器的 WZ.XML 覆蓋客戶端 WZ
❌ 刪除後重新匯入節點
❌ 使用非 GMS (old) 加密格式儲存
```

### NPC 腳本基本模板

```javascript
var status = 0;

function start() {
    status = -1;
    action(1, 0, 0);
}

function action(mode, type, selection) {
    if (mode == -1) {
        cm.dispose();
    } else if (mode == 1) {
        status++;
    } else {
        status--;
    }
    
    if (status == 0) {
        cm.sendNext("你好！選擇一個選項:");
    } else if (status == 1) {
        cm.sendSimple("#b#L0#購買道具#l\n#L1#出售道具#l\n#L2#離開#l");
    } else if (status == 2) {
        if (selection == 0) {
            // 購買邏輯
            cm.dispose();
        } else if (selection == 1) {
            // 出售邏輯
            cm.dispose();
        } else {
            cm.sendNext("歡迎再次光臨！");
            cm.dispose();
        }
    } else {
        cm.dispose();
    }
}
```

### NPC 腳本源相容性問題 (2026-03 新發現)

**問題**：
- 不同音源（Cosmic、Kinoko、HeavenMS）的 NPC 腳本解析器版本不同
- 較舊的腳本在新解析器可能不相容

**原因**：
- `importPackage` 和 `SavedLocationType` 處理差異
- JDK 7+ 已廢棄 `importPackage`

**解決方案**：
1. 使用普遍支援的 `cm.` 函數
2. 避免使用特定音源專有的 Package 路徑
3. 避免 `importPackage`（改用直接引用）
4. 先在本地測試再部署到正式伺服器
