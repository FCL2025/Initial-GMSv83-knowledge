## 🆕 Kinoko NPC 腳本源碼與相容性分析 (RaGEZONE, 4週前)

**來源**: https://forum.ragezone.com/threads/kinoko-npc-scripts.1261538/

### 核心問題：不同音源 NPC 腳本相容性

**問題描述**：
不同模擬器（Cosmic、Kinoko、HeavenMS）的 NPC 腳本存在相容性差異，主要原因：
1. 腳本解析器版本不同
2. API 物件名稱不一致
3. 較舊的腳本在新解析器可能不相容

### Kinoko 腳本源碼特色

**Kinoko** 是一個較新的 MapleStory 模擬器專案，其 NPC 腳本有以下特點：

```javascript
// Kinoko NPC 腳本範例結構
var status = 0;

function start() {
    // Kinoko 使用 cm 物件（與 Cosmic 相容）
    cm.sendNext("Hello, traveler!");
}

function action(mode, type, selection) {
    if (mode == -1) {
        cm.dispose();
    } else {
        if (mode == 1) status++;
        else status--;
        
        if (status == 1) {
            cm.sendNext("Welcome to Kinoko!");
        } else {
            cm.dispose();
        }
    }
}
```

### NPC 腳本相容性對照表

| 音源 | 腳本引擎 | API 物件 | ES6 支援 | importPackage |
|------|---------|---------|---------|--------------|
| **Cosmic** | Nashorn | `cm` | ✅ | ❌ |
| **Kinoko** | Nashorn | `cm` | ✅ | ❌ |
| **IntransigentMS** | Nashorn | `cm` | ✅ | ❌ |
| **HeavenMS** | JDK 6 Rhino | `cm` | ❌ | ✅ |

### 相容寫法（跨音源通用）

```javascript
// ✅ 推薦寫法（Cosmic + Kinoko + IntransigentMS）
var MapleCharacter = Packages.client.MapleCharacter;
cm.getPlayer();

// ❌ 舊寫法（只相容 HeavenMS）
importPackage(Packages.client);
cm.getPlayer();
```

### 實用建議
- 開發新 NPC 腳本時，使用 `Packages.<package>.<Class>` 語法
- 避免 `importPackage()`，因為 Nashorn 不支援
- 測試時在多個音源環境中驗證腳本

## 🆕 MapleStory v0.40b C# 客戶端實現 (RaGEZONE, 2025-05)

**來源**: https://forum.ragezone.com/threads/maplestory-v0-40b-hosting-help.1244683/

### v0.40b 與 v83 的差異

| 項目 | v83 | v0.40b |
|------|-----|--------|

| 客戶端架構 | C++ | C# |

## 🆕 RustMS 2026年3月重啟：非 JVM 登入伺服器 (GitHub, 2026-03)

**GitHub**: https://github.com/neeerp/RustMS

### 重啟背景
- 專案最初為學習 Rust 而建立，後被放棄
- 2026年3月重啟，使用 AI coding assistant 輔助開發
- 目前僅包含基本可用的 **Login Server**（監聽 localhost:8484）

### RustMS 技術架構

```
rust-ms-login (entry point)
  → 監聽 localhost:8484
  → 處理 AES handshake
  → 管理客戶端連線的 IO 迴圈

net::io (網路核心)
  → 封包讀寫、加密/解密
  → 封包委派處理
  → Session 狀態管理

net::packet (封包處理)
  → handle/ - 封包處理器
  → build/ - 封包構建器

crypt (加密)
  → MapleAES (OFB 模式)
  → Shanda 加密（第二層）
  → bcrypt (密碼雜湊)

db (資料庫)
  → Diesel ORM
  → PostgreSQL
  → 帳號、角色等 entity
```

### RustMS 加密實現亮點

**MapleAES - OFB 模式**：
```rust
// MapleAES 負責封包加密的核心結構
// 特點：
// - 追蹤 send/recv 的 nonce（遞增向量）
// - 支援 OFB 模式的 AES 加密
// - 自動處理 IV 變異
```

**bcrypt 密碼雜湊**：
```
密碼不再明文傳輸
伺服器端使用 bcrypt 驗證
更安全的密碼存儲方案
```

### RustMS vs Java 模擬器

| 特性 | RustMS | Java 模擬器 (Cosmic/HeavenMS) |
|------|--------|-------------------------------|
| 語言 | Rust | Java |
| 記憶體效率 | ✅ 極佳（無 GC） | ⚠️ JVM 開銷 |
| 執行速度 | ✅ 優秀 | ⚠️ JIT 需預熱 |
| 生態完整度 | ⚠️ 僅 Login Server | ✅ 全功能 |
| 學習資源 | ⚠️ 匱乏 | ✅ 豐富 |
| 維護難度 | ⚠️ Rust 陡峭 | ✅ Java 簡單 |
| 社群 | ⚠️ 初期 | ✅ 活躍 |

### RustMS 研究價值
1. **清晰的代碼結構** - Rust ownership 模型強制良好的架構設計
2. **完整加密實現** - MapleAES + Shanda + bcrypt 全部開源

## 🆕 Maple_Pshark - v83 專用封包 Logger 實測驗證 (GitHub)

**GitHub**: https://github.com/obstriker/Maple_Pshark

### 專案現況（2026-03 驗證）
- **Mapleroyals packet logger** - Maplestory v83 專用封包截獲器
- Hook 加密函數，直接截獲明文
- 支援封包修改和注入
- 規則化封包操作/過濾

### 四大核心功能

| 功能 | 說明 |
|------|------|
| `1.` Intercepting Recv messages | 攔截客戶端接收封包 |
| `2.` Modify messages | 修改封包內容並重新注入 |
| `3.` Injecting packets | 支援 Recv 和 Send 方向的注入 |
| `4.` Rule based filtering | 自訂規則過濾特定封包 |

### Maple_Pshark vs MaplePacketPuller

| 工具 | 類型 | 特色 |
|------|------|------|
| Maple_Pshark | 即時截獲器 | Hook 加密函數，無需解密鑰 |
| MaplePacketPuller | 離線分析器 | 從 pcap 分析封包結構 |

## 🆕 IntransigentMS - Nashorn ES6 NPC 腳本源碼 (GitHub)

**GitHub**: https://github.com/NoetherEmmy/intransigentms-scripts

### 腳本引擎：Nashorn + ECMAScript 6

```javascript
// IntransigentMS NPC 腳本範例
const MapleCharacter = Java.type("net.sf.odinms.client.MapleCharacter");

// ES6 寫法（const + Arrow functions）
const isBeginner = (player) => player.getJobId() === 0;

// 調試輸出
print("DEBUG: NPC script started");
```

### IntransigentMS 腳本特色
- **Nashorn 引擎**：Java 8+ 內建的 JavaScript 引擎
- **ECMAScript 6**：支援 const、let、Arrow Functions、Template Literals
- **Java.type()**：Nashorn 專用語法，用於引用 Java 類別
- **不使用 importPackage()**：與舊版 Rhino 不同

### ES6 vs 舊版 JavaScript 語法對照

| 功能 | 舊版 Rhino | Nashorn ES6 |
|------|-----------|-------------|
| 變數宣告 | `var` | `const`, `let` |
| 函數 | `function() {}` | `() => {}` |
| 字串拼接 | `"a" + b` | `` `a${b}` `` |
| 類別引用 | `importPackage(Packages.x)` | `Java.type("x")` |

## 🆕 maplelib Go 加密庫：雙金鑰機制核心原理

**來源**: https://pkg.go.dev/github.com/defaultmagi/maplelib

### 四個 IV 的意義

| IV | 擁有者 | 用途 |
|----|--------|------|
| 客戶端 SendIV | 客戶端 | 加密發送到伺服器的封包 |
| 客戶端 RecvIV | 客戶端 | 解密從伺服器收到的封包 |
| 伺服器 SendIV | 伺服器 | 加密發送到客戶端的封包 |
| 伺服器 RecvIV | 伺服器 | 解密從客戶端收到的封包 |

### MapleCrypto 頭部混淆公式

**加密端（getPacketHeader）**：
```java
int verify = (opcode ^ (iv & 0xFF)) + ((iv >> 8) & 0xFF) * 0x100;
verify = (verify << 3) | (verify >>> 5);  // rotate left 3 bits
verify &= 0xFFFF;
```

**解密端（retrieveOpcode）**：
```java
int verify = (header[0] & 0xFF) | ((header[1] << 8) & 0xFF00);
verify = (verify >>> 3) | ((verify << 5) & 0xFFFF);
return (verify - ((iv >> 8) & 0xFF) * 0x100) ^ (iv & 0xFF);
```
