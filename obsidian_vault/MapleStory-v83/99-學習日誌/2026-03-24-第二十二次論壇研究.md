### 封包加密三層架構 (v83)
```
Layer 1: MapleCrypto (頭部混淆)
  - 4-byte header，含 opcode + length
  - verify = (opcode ^ iv[0]) + (iv[1] << 8)
  - verify = (verify << 3) | (verify >>> 5)

Layer 2: Shanda Encryption (對稱加密)
  - v83 仍在用，已於 v118+ 移除

Layer 3: AES-OFB (主加密)
  - 256-bit key，版本決定的靜態金鑰
```

### Hello Packet 結構（永遠明文）

**客戶端 → 伺服器**：
```
偏移  長度   內容
0x00  2      版本號 (0x0053 = 83)
0x02  n      版本字串 ("83")
n+2   4      SendIV
n+6   4      RecvIV
n+10  2      地區標誌
n+12  2      Unknown
```

**為什麼第一個封包是純文字**：
- MapleStory 使用「加密握手」流程
- 雙方需要用相同的 IV 初始化 AES-OFB 加密器
- 若加密，雙方都無法解密對方的第一個訊息

### 完整登入流程封包序列 (10步)
```
1. C→S Hello Packet (plaintext)
2. S→C Hello Packet (plaintext)
3. C→S LOGIN_PASSWORD (encrypted) - Opcode 0x01
4. S→C LOGIN_STATUS (encrypted) - Opcode 0x00
5. C→S SERVERLIST_REQUEST (encrypted) - Opcode 0x06
6. S→C SERVERLIST (encrypted) - Opcode 0x0C
7. C→S CHARLIST_REQUEST (encrypted) - Opcode 0x07
8. S→C CHARLIST (encrypted) - Opcode 0x03
9. C→S CHAR_SELECT (encrypted) - Opcode 0x08
10. C→S PLAYER_DROP (encrypted) - Opcode 0x0B
```

## 📌 Opcode 結構

### MapleCrypto Header 加密/解密公式

**加密端（發送）**：
```java
public byte[] getPacketHeader(int opcode, int length, int iv) {
    int verify = (opcode ^ (iv & 0xFF)) + ((iv >> 8) & 0xFF) * 0x100;
    verify = (verify << 3) | (verify >>> 5);
    verify &= 0xFFFF;
    return new byte[] {
        (byte) (verify & 0xFF),
        (byte) ((verify >>> 8) & 0xFF),
        (byte) (length & 0xFF),
        (byte) ((length >>> 8) & 0xFF)
    };
}
```

**解密端（接收）**：
```java
public int retrieveOpcode(byte[] header, int iv) {
    int verify = (header[0] & 0xFF) | ((header[1] << 8) & 0xFF00);
    verify = (verify >>> 3) | ((verify << 5) & 0xFFFF);
    return (verify - ((iv >> 8) & 0xFF) * 0x100) ^ (iv & 0xFF);
}
```

### CClientSocket::ProcessPacket 封包分發流程

```
CClientSocket::ProcessPacket()
    ↓ 根據 MapleCrypto header 解密
    ↓ 提取 opcode (2 bytes, little endian)
    ↓ switch(opcode) 分發
    → CLogin::OnPacket()    // 登入畫面
    → CField::OnPacket()     // 遊戲中
    → CShop::OnPacket()      // 商場
```

### IDA 分析 Opcode 進階技巧

**使用 IDA 找到封包處理函數**：
1. 找 `CClientSocket::ProcessPacket` - 客戶端接收封包的總入口
2. 從這裡根據 header 分發到 `CField::OnPacket` 或 `CLogin::OnPacket`
3. 這就是為什麼有 opcode 清單就能輕鬆找到客戶端接收封包的處理函數

**AOB (Array of Bytes) 搜尋**：
- 在 IDA 中使用 Hex View 搜尋已知 opcode 的 bytes 模式
- 可找到 opcode 表的記憶體位置

### v83 封包長度識別技巧

```
為什麼相同操作產生的封包每次長度都不同？
原因：IV (初始化向量) 每次都會改變
每次加密時，IV 都會遞增/更新
導致相同 opcode 的封包看起來長度不同

識別封包類型的技巧：
- 長度固定 (如每次 32 bytes) → 可能是 MOVE_PLAYER (0x14)
- 包含可讀字串 → 可能是 CHAT
- 第一個 byte 為 0x00 → 伺服器回應封包
```

```javascript
                cm.sendOK("抱歉，收購服務暫時關閉。");
                cm.dispose();
            } else {
                cm.dispose();
            }
        }
    }
```

```javascript

        cm.dispose();

            status++;
        else

                cm.gainMeso(-item[1]);
                cm.gainItem(item[0], item[2]);
                cm.sendOk("購買成功！");
            } else {
                cm.sendOk("金幣不足！");
            }
            cm.dispose();
        }
    }
}
```

| NPC 腳本不相容 | 使用普遍 `cm.` 函數，避免專有Package |
| 不同音源解析器版本不同 | 先在本地測試再部署到正式伺服器 |

**最佳實踐**：
- 使用普遍支援的 `cm.` 函數
- 避免使用特定音源專有的 Package 路徑
- 避免 `importPackage`（JDK 7+ 已廢棄）
- 先在本地測試再部署到正式伺服器
