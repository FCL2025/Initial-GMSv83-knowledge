## 🆕 MapleRef Wiki — Opcode 結構核心概念

**URL**: https://mapleref.fandom.com/wiki/Packets

### 核心定義
> "Actions that are handled by packets on MapleStory are defined by an identifier called an Operation Code (opcode for short), which is the first 2 bytes of the packet (a short)"

### Send vs Receive Opcode 區分

| 方向 | 說明 | Opcode 集合 |
|------|------|------------|
| **S→C (Send)** | 伺服器發送到客戶端 | Server Opcode Set |
| **C→S (Recv)** | 客戶端發送到伺服器 | Client Opcode Set |

**重要**：同一個 Opcode 數字在不同方向代表**完全不同**的含義！

### MapleRef 使用方式
```
1. 前往 https://mapleref.fandom.com/wiki/Packets
2. 選擇目標版本 (v83)
3. 查看 Send (S→C) 或 Recv (C→S) 封包結構
4. 配合 Wireshark 抓包分析
```

## 🆕 IDA 分析Opcode — CClientSocket::ProcessPacket 流向 (RaGEZONE, 2019)

**URL**: https://forum.ragezone.com/threads/how-does-one-find-a-structure-of-a-packet-via-ida.1162654/

### 所有客戶端接收封包的入口點
```
CClientSocket::ProcessPacket
    ↓
根據 MapleCrypto header 解密
    ↓
提取 opcode (2 bytes, little endian)
    ↓
switch(opcode) 分發
    ↓
CLogin::OnPacket()    // 登入畫面
    或
CField::OnPacket()    // 遊戲中
```

### 封包處理函數命名慣例

| 函數前綴 | 狀態 | 說明 |
|---------|------|------|
| `CLogin::On*` | 登入畫面 | 登入相關封包處理 |
| `CField::On*` | 遊戲中 | 遊戲內封包處理 |
| `CShop::On*` | 商場 | 商場相關封包處理 |

### 為什麼知道 Opcode 就能找到處理函數
- `CClientSocket::ProcessPacket` 內有一個大 switch 表
- 每個 case 對應一個 opcode
- case 內直接調用具體的 OnPacket 處理函數
- 因此知道 opcode 就能在 IDA 中快速定位函數

### 找封包結構的步驟
```
1. IDA 中搜尋 CClientSocket::SendPacket
2. 找到 packet header（例：0x1C 為 Inventory Operation）
3. 追蹤 decode 函數調用
4. 確認 packet structure 是否改變
```

### 編譯設定
```

2. 在專案屬性加入：
   $(SolutionDir)Common;%(AdditionalIncludeDirectories)
3. 必須編譯為 x86（x64 會有 detour linker errors）
```

### 兩種注入方式

| 方式 | 設定 | 說明 |

| **ijl15 proxy 自動注入** | 定義 `MAPLE_INJECT_USE_IJL` | MapleStory 啟動時自動注入，需將 DLL 改名為 `ijl15.dll` |
| **GenericLauncher 手動注入** | 普通注入 | 啟動 MapleStory 並注入 `LEN.dll` |

### 架構說明

Common 專案：所有 Windows API hooks（非版本特定）

## 🆕 MapleSolaxiaV2 WZ 修改黃金法則

**來源**: MapleSolaxiaV2 README

### WZ 修改最重要原則

> **"DO NOT USE the server's XMLs for reimporting into the client's WZ, it WILL generate bugs."**

**正確流程**：
1. HaRepacker 4.2.4 開啟 **GMS (old)** 加密的客戶端 WZ
2. **使用 UI 直接修改數值**（禁止刪除/重新導入節點！）
3. 儲存 WZ
4. 在 HaRepacker 中選擇 **"Private Server..."** 導出
5. 將導出的 XML 覆蓋到伺服器的 WZ.XML 目錄

### 為什麼不能直接用伺服器 XML？

| 問題 | 原因 |
|------|------|
| WZ 結構差異 | 客戶端和伺服器 WZ 結構不完全一致 |
| 加密格式不同 | 客戶端使用 GMS (old)，伺服器使用 Private Server 格式 |
| CRC 校驗問題 | 錯誤的 WZ 會導致客戶端崩潰 |

## 🆕 NPC 腳本除錯 — 常見崩潰原因與解決方案 (RaGEZONE, 2026-03)

**來源**: https://forum.ragezone.com/threads/need-help.1262639/

### 問題：NPC 對話後客戶端崩潰

**可能原因**：
1. `cm.dispose()` 在狀態機中途被呼叫
2. `cm.sendNext()` 的文字包含無效的格式標籤
3. 選項 (`#L0#...#l`) 數量與 `selection` 處理不匹配
4. 嘗試讀取不存在的玩家屬性

### 解決方案

```javascript
// 檢查項 1：確保 dispose() 只在對話結束時呼叫
if (mode == -1 || (mode == 0 && status == lastStatus)) {
    cm.dispose();
    return;
}

// 檢查項 2：確保格式標籤正確配對
// #b 藍色  → 需 #k 重置
// #r 紅色  → 需 #k 重置
// #e 粗體  → 需 #n 重置

// 檢查項 3：選項數量與處理匹配
// 若有 3 個選項，selection 範圍應為 0~2
if (selection < 0 || selection >= options.length) {
    cm.dispose();
    return;
}
```
