## 四十六、封包加密機制深度解析（最新研究）

### 46.1 v83 封包加密架構

**重要發現：v83 使用兩層加密！**

MapleStory v83 的封包加密流程（客戶端 ↔ 伺服器）：

```
【發送方向：客戶端 → 伺服器】
客戶端發送前：
1. 先用 Shanda (MapleCustomEncryption) 加密
2. 再用 AES-OFB 加密
3. 發送

伺服器接收後：
1. 先用 AES-OFB 解密
2. 再用 Shanda 解密
3. 處理數據

【接收方向：伺服器 → 客戶端】
反向同理，兩端各自維護 IV（初始向量）
```

**第一個封包（握手封包）是明文！**

所有版本（包括 v83）的第一個封包都是 **未加密** 的，用於交換 AES 金鑰。

### 46.2 握手流程（Handshake）

```
1. 客戶端 → 連接到伺服器端口（8484 / 7575）
2. 伺服器 → 發送【明文】握手封包，內容：
   - 隨機生成的 AES Key（用於加密發送方向）
   - 隨機生成的 AES IV（用於解密接收方向）
   - 伺服器期望的客戶端版本
3. 客戶端 → 發送【明文】握手回應
   - 確認版本
   - 客戶端的 Key/IV（用於伺服器加密發送方向）
4. 此後所有封包都加密
```

**為何握手封包是明文？**
- TCP 連線剛建立時，雙方還沒有協商加密金鑰
- 需要用明文交換金鑰
- 這也意味著：如果有人在線路上監聽，可以看到金鑰

### 46.3 兩把鑰匙（Two Keys）

MapleStory 使用**兩組不同的 Key/IV**：

| 方向 | 客戶端加密 | 伺服器解密 |
|------|-----------|-----------|
| 客戶端 → 伺服器 | 客戶端的 Key | 伺服器用收到的客戶端 Key 解密 |

| 方向 | 伺服器加密 | 客戶端解密 |
|------|-----------|-----------|
| 伺服器 → 客戶端 | 伺服器的 Key | 客戶端用收到的伺服器 Key 解密 |

**實際意義：**
- 如果要攔截並解密流量，需要知道雙方的 Key/IV
- 兩組 Key 完全獨立，無法從一組推導另一組

### 46.4 AES-OFB vs Shanda 加密

**v83 適用的加密層：**

| 加密層 | 說明 | 用於版本 |
|--------|------|---------|
| Shanda (MapleCustomEncryption) | 每次 IV 都會更新（遞增/XOR）| v83 及更早 |
| AES-OFB | AES-128-OFB 模式，Block 加密 | v83 |
| 混合 | Shanda + AES-OFB 兩層 | v83（客戶端→伺服器）|

**Shanda 加密函數關鍵行為：**
- IV 每次操作後都會改變（防止重放攻擊）
- 如果移除 Shanda，IV 不會更新，看起來「正常」但實際上不安全
- v83 的 Shanda 加密是每次遞增一個計數值

**注意：**
- 新版（GMS v140+）移除了 Shanda，只用 AES
- v83 是最後一個大量使用 Shanda 的版本

### 46.5 封包結構詳解

**v83 封包格式（小端序）：**

```
[Length: 4 bytes][Opcode: 2 bytes][Payload: N bytes]

Length = 整個封包的長度（包含 Length 自己）
Opcode = 2 bytes，小端序（Little Endian）
Payload = 實際數據
```

**Opcode 解讀範例：**

```
十六進位視角（TCP Stream）：
01 00 0D 00 61 64 6D 69 6E 00 [32 bytes hash]

拆分：
01 00 00 00 00        # Length = 0x00000001? 不对，这是第一個封包的特殊格式
                      # 實際上第一個封包有特殊格式

# 更常見的封包：
14 00 2C 01 00 00 00 00 00 00 64 00 ...
     ^^-- Opcode = 0x0114 = 276 = SPAWN_MONSTER (little-endian)
```

**常見 Decode 函數對照（IDA 分析）：**

| IDA CInPacket::Decode | 對應 mplew::write | 資料類型 |
|----------------------|-------------------|---------|
| `Decode1()` | `write()` | byte |
| `Decode2()` | `writeShort()` | short (2 bytes) |
| `Decode4()` | `writeInt()` | int (4 bytes) |
| `DecodeStr()` | `writeMapleAsciiString()` | 字串（長度前綴）|
| `DecodeBuffer(x, 8)` | `writeLong()` | long (8 bytes) |
| `DecodeBuffer(x, 13)` | `writeAsciiString(str, 13)` | 固定13字節 |

## 四十七、IDA 分析實戰 — 找到客戶端封包處理函數

### 47.1 客戶端封包流向（v83）

```
【收到的封包（伺服器 → 客戶端）】
CClientSocket::ProcessPacket
    ↓
根據 Opcode 分發
    ↓
CField::OnPacket      (Opcode >= 0x100, 地圖相關)
CLogin::OnPacket      (Opcode < 0x100, 登入相關)
CWvsContext::OnPacket (Opcode < 0x100, 通用)
    ↓
CInPacket::Decode1/2/4/Str 讀取數據

【發送的封包（客戶端 → 伺服器）】
mplew::writeShort(opcode)
mplew::writeInt/Str/...
    ↓
加密（Shanda + AES-OFB）
    ↓
發送
```

### 47.2 IDA 分析流程（找客戶端發送封包）

**目標：找到 `LOGIN_REQUEST` 的發送位置**

1. **搜尋字串**
   ```
   ALT+T → 搜尋 "LOGIN" 或 "username"
   ```

2. **找到交叉引用**
   ```
   X（交叉引用）→ 查看所有引用此字串的函數
   ```

3. **找到 CInPacket 建構函數**
   ```
   CInPacket::CInPacket(byte*, int)
   或
   CInPacket::CInPacket()
   ```

4. **分析封包發送流程**
   ```
   mplew::writeShort(0x01)     // LOGIN_REQUEST opcode
   mplew::writeMapleAsciiString(id)
   mplew::writeMapleAsciiString(password)
   ```

### 47.3 IDA 分析實例 — v83 封包 opcode 結構

**v83 的 packets 特色（RaGEZONE 網友分享）：**

```
在 IDA 中找到 CInPacket::Decode 的調用：

Decode1()      → mplew.write()       → 1 byte
Decode2()      → mplew.writeShort()   → 2 bytes
Decode4()      → mplew.writeInt()     → 4 bytes
DecodeStr()    → mplew.writeMapleAsciiString() → 字串
DecodeBuffer(x, 8)  → mplew.writeLong()  → 8 bytes
DecodeBuffer(x, 13)  → mplew.writeAsciiString(str, 13) → 固定13字節
```

**v83 的 packets 結構普遍比較完整**，與新版本相比，更容易在 IDA 中分析。

### 47.4 找 Opcode Handler 的另一種方法

**步驟：**

1. 在 IDA 中 `ALT+T` 搜尋 `CClientSocket`
2. 找到 `ProcessPacket` 函數
3. `F5` 反編譯
4. 分析 `switch (opcode)` 語句

**實例：`if (a2 < 177 || a2 > 275)` 這類判斷**

根據 opcode 值範圍分發到不同的 handler：
- `a2 < 114` → `CWvsContext::OnPacket`
- `114 <= a2 < 140` → 某個 handler
- `276 <= a2 <= 302` → Monster handler（怪物相關）
- `303 <= a2 <= 310` → 另一個 handler

## 四十八、WZ 資源最新動態（2026-03）

### 48.1 v117 髮型/臉型導入 v83 流程（RaGEZONE 2023）

**需求工具：**
- v117 MapleStory 客戶端（WZ 來源）
- v83 WZ 檔案（目標）
- HaRepacker 編輯 v83 WZ
- OllyDbg（可選，用於調試）

**流程摘要：**

1. **使用 HaRepacker 打開 Character.wz（v83）**
   - 加密選擇：`GMS (old)`
   - 找到 `Hair.img` 或 `Face.img`

2. **從 v117 提取資源**
   - 用 HaRepacker 打開 v117 的 Character.wz
   - 找到對應的 hair/face 節點

3. **複製粘貼到 v83**
   - 選擇 v117 的 hair/face 節點
   - 粘貼到 v83 的 Character.wz

4. **處理依賴**
   - `String.wz` 需要對應的 hair/face 名稱
   - `Sound.wz` 可能需要背景音樂

**⚠️ 注意：**
- v117 和 v83 的 .img 結構可能不完全相同
- 某些 hair/face 依賴高版本的特效或著色器
- 建議先在本地測試

**參考教程：**
- 普通版：forum.ragezone.com/threads/how-to-import-hairs-and-faces-from-v117-to-v83.1215383/

### 48.2 Reactor.wz 修改實例 — 無延遲箱子（HPQ）

**HPQ（Horned Quay PQ）箱子延遲修改：**

1. 用 HaRepacker 打開 `Reactor.wz`
2. 找到對應的 reactor（如 `9200002.img` — 箱子）
3. 修改屬性：
   - `info/regenPattern` — 重生模式
   - `info/regenMap` — 重生地圖
   - `info/breakType` — 破壞類型

**無延遲設定：**
- `delay` = 0
- `regenTime` = 0
- 直接在腳本中控制重生

### 48.3 Quest.wz 修改 — 等級限制移除（更新版）

**2025 年最新方法：**

```
1. 用 HaRepacker 打開 Quest.wz
2. 導航到 Quest/Check.img.xml
3. Tools → Export selection → XML → Classic
4. 用 Notepad++ 搜尋 lvmax 屬性
5. 移除 lvmax 或設為更高值
6. Import XML 回 HaRepacker
7. 存檔
```

**常見問題：**
- 如果移除所有 lvmax，導致高等級任務被低等級角色接取
- 建議只修改必要的任務

### 48.4 etc.wz 修改 — Ban Message 惡搞

**步驟：**
1. 用 HaRepacker 打開 `etc.wz`
2. 找到 `blockreason.img`
3. 選擇任意 number（12 是 A/B 訊息）
4. 修改 msg 為你想要的內容
5. 存檔

**⚠️ 注意：** 這只用於私人伺服器娛樂，正式伺服器請勿使用。

## 四十九、封包加密關鍵原始碼資源

### 49.1 maplelib（Go 語言實現）

**GitHub：** https://pkg.go.dev/github.com/defaultmagi/maplelib

**完整加密流程：**

```go
// 客戶端加密（發送方向）
func (c *Client) encrypt(data []byte) ([]byte, error) {
    // 1. Shanda 加密
    encrypted := shanda.Encrypt(data, c.sendIV)
    // 2. AES-OFB 加密
    final := aes.OfbEncrypt(encrypted, c.sendKey, c.sendIV)
    return final, nil
}

// 伺服器解密（接收方向）
func (s *Server) decrypt(data []byte) ([]byte, error) {
    // 1. AES-OFB 解密
    decrypted := aes.OfbDecrypt(data, s.recvKey, s.recvIV)
    // 2. Shanda 解密
    final := shanda.Decrypt(decrypted, s.recvIV)
    return final, nil
}
```

**握手封包交換：**

```go
// 伺服器發送握手
func (s *Server) sendHandshake() error {
    // 生成隨機 Key/IV
    sendKey := generateRandomKey()   // 16 bytes
    sendIV := generateRandomIV()     // 16 bytes
    
    // 發送明文握手（包含客戶端要用來解密伺服器發送的封包）
    packet := append(sendKey, sendIV...)
    return s.conn.Write(packet)
}
```
