## 🆕 MapleEzorsia V2 — GitHub README 深度研究

**來源**: https://github.com/444Ro666/MapleEzorsia-v2

### 設計哲學（原文）

> "The original goal of this DLL is to allow an old 4:3 aspect ratio game to work on 16:9 aspect ratio modern monitors while preserving the original gameplay experience as much as possible."

**設計哲學**：
- 適用於任何客戶端或 localhost（包括原廠安裝的預設客戶端）
- 不需要刪除原廠任何檔案，僅需放入 dinput8.dll 即可
- 不需要額外的 hackshield 或 patcher 刪除

### 技術實現（原文）

> "This DLL modifies addresses in a default, packed, v83 MapleStory client fresh from installation to enable the game to run; as well as changes the game window and canvas resolution to HD."

**核心技術**：
- 修改原廠封裝客戶端中的記憶體位址
- 將遊戲視窗和畫布解析度改為 HD
- 大多數遊戲內元件邊界也會隨自訂解析度縮放

### 編譯平台要求

| 要求 | 說明 |
|------|------|
| 編譯平台 | **x86（32-bit）**，不支援 x64 |
| 工具链 | C++ 編譯器、Visual Studio |
| 參考設置 | https://www.mediafire.com/view/9ssrqg8eiwbh0k1/buildsettings2.png |
| Ezorsia 專用 | https://www.mediafire.com/view/6xk7b9bf8qv3gda/buildsettingsEzorsia.png |

### config.ini 重要警告（原文）

> "try to avoid extending the comments (behind ;comment) inside of it or INIReader will be unable to read from it and crash you"

**建議**：不要延伸 config.ini 中的注釋（`;` 後面的內容），否則 INIReader 無法正確讀取導致崩潰。

### 第三方 WZ 編輯相容性（原文）

> "I suggest doing all your edits in your own EzorsiaV2_UI.wz or MapleEzorsiaV2wzfiles.img (if using .IMG file) to avoid WZ conflicts with the base game, and to ensure you can keep track of them."

建議將所有 WZ 編輯放在 `EzorsiaV2_UI.wz` 或 `MapleEzorsiaV2wzfiles.img` 中，以避免與原廠 WZ 衝突並便於追蹤。

### 4G Flag Admin 提權移除重要發現（原文）

**問題**：編輯 `asInvoker` 後，客戶端仍然要求管理員權限

**根本原因**：當執行檔命名為預設名稱 `MapleStory.exe` 時，Windows UAC 會繞過自訂的 `asInvoker` 設定

**解決方案**：
1. 使用 Hendi's Hex Editor 工具包編輯 `asInvoker` 部分
2. **將執行檔改名為 `MapleStory2.exe` 或其他名稱**
3. Windows 就會正確應用 `asInvoker` 設定，不再要求管理員權限

## 🆕 MapleClientEditTemplate — C++ 客戶端修改框架（oung/Minimum Delta）

**GitHub**: https://github.com/oung/MapleClientEditTemplate

### 設計理念

> "An increasingly generic and comprehensive MapleStory client editing framework."

**適用範圍**：
- 適用於任何版本（版本特定的 anticheat 和 CRC bypass 需要調整）
- 所有 Windows library hooks 已抽象化，可在 `commonconfig.h` 中開關

### v83 特殊設定

對於 v83 等**未虛擬化的 localhost**：
- 可以**禁用 WinSock hooks**
- 直接進行記憶體編輯（因為不需要等待客戶端解包）

### 編譯設定（重要）

```
1. 將 detours.lib 放入 Debug 資料夾
2. 在專案屬性加入：
   $(SolutionDir)Common;%(AdditionalIncludeDirectories)
3. **必須編譯為 x86**（x64 會有 detour linker errors）
```

### 兩種注入方式

| 方式 | 設定 | 說明 |
|------|------|------|
| **ijl15 proxy 自動注入** | 定義 `MAPLE_INJECT_USE_IJL` | MapleStory 啟動時自動注入，需將 DLL 改名為 `ijl15.dll` |
| **GenericLauncher 手動注入** | 普通注入 | 啟動 MapleStory 並注入 `LEN.dll` |

### 架構說明

```
Common 專案：所有 Windows API hooks（非版本特定）
  - 在 injection 後立即執行
  - 觸發時機：CreateMutexA hook 觸發 MainFunc
  - 建議將所有 hooking/memory editing 放在 MainFunc 中
  - MapleStory function hooks：版本特定（尚未加入模板）
```

| 功能 | 說明 |

### 核心功能列表

## 🆕 MapleRef Wiki — Opcode 核心概念研究

**URL**: https://mapleref.fandom.com/wiki/Packets

### 核心原則

> "Actions that are defined by an identifier called an Operation Code (opcode for short), which is the first 2 bytes of the packet (a short)"

**重要**：在 v83 中 opcode 是封包的第一個 2 bytes（short）。

### Send vs Receive Opcode 區分

| 方向 | 說明 |
|------|------|
| **S→C (Send)** | 伺服器發送到客戶端 |
| **C→S (Recv)** | 客戶端發送到伺服器 |

**重要**：同一個 Opcode 數字可能代表完全不同含義，視方向而定！

### MapleRef 使用方式

```
1. 前往 https://mapleref.fandom.com/wiki/Packets
2. 選擇目標版本 (v83)
3. 查看 Send (S→C) 或 Recv (C→S) 封包結構
4. 配合 Wireshark 抓包分析
```

## 🆕 botnored — MapleStory v83 Classes 結構

**URL**: https://botnored.weebly.com/blog/maplestory-classes-v83

### 重要原則

> **第一個發送/接收的封包永遠是純文字（未加密）**
> MapleStory 封包透過 TCP Socket 傳輸

### 客戶端主要類別

| 類別 | 功能 |
|------|------|
| `CWvsContext` | 世界狀態管理（角色資料、伺服器列表） |
| `CClientSocket` | 網路連接管理（Send/Recv） |
| `CField` | 當前地圖/遊戲場景處理 |
| `CLogin` | 登入畫面處理 |

### 封包流向

```
CClientSocket::ProcessPacket()
    ↓
根據 Header 分發
    ↓
CField::OnPacket() 或 CLogin::OnPacket()
    ↓
處理具體 Opcode
```

## 🆕 MapleStory v83 加密核心概念

**來源**: https://pkg.go.dev/github.com/defaultmagi/maplelib

### 雙金鑰機制

| 金鑰 | 用途 | 方向 |
|------|------|------|
| **Send Key** | 客戶端加密 / 伺服器解密 | C → S |
| **Recv Key** | 伺服器加密 / 客戶端解密 | S → C |

### 第一個封包（Hello Packet）

- **永遠是明文**（未加密）
- 用於客戶端和伺服器交換 IV
- 包含版本號和 IV 值

### AES Handshake 流程

```
1. C→S: Hello Packet (plaintext)
   - 客戶端告知版本、SendIV、RecvIV

2. S→C: Hello Packet (plaintext)
   - 伺服器確認版本、回應 IV

3. 之後所有封包：MapleCrypto + Shanda + AES-OFB 加密
```

### v83 各版本加密變化

| 版本 | 加密方式 | 備註 |
|------|---------|------|
| v83~v117 | Shanda + AES-OFB | ✅ 仍在使用 |
| v118+ | 僅 AES-OFB | Shanda 已移除 |
