### 完整抓包工作流程 (2026 最新)

```
Step 1: Wireshark 確認 TCP 連線
  tcp.port == 7575 || tcp.port == 8484
  ↓ 確認 Hello Packet 交換

Step 2: Maple_Pshark 即時截獲
  Hook 加密函數，直接取得明文

  https://mapleref.fandom.com/wiki/Operation_Code
  確認封包含義
```

### botnored v83 封包核心原則 (botnored 2022-2026)

> **"The first packet sent / received will always be in plain-text (not encrypted)."**
> MapleStory packets are performed though a TCP socket.

- 第一個封包（Hello Packet）永遠是明文
- 之後所有封包經過：Shanda Encryption → AES-OFB → MapleCrypto Header
- v83 封包 Little Endian 格式（位元組順序顛倒）

### v83 封包加密三層結構

```
┌────────────────────────────────────────┐
│ 1. MapleCrypto Header (4 bytes)        │
│    - 包含長度 + 混淆 opcode             │
│                                           │
│ 2. Shanda Encryption (v83~v117)        │
│    - 在 AES 之前應用                     │
│    - v118+ 已移除                        │
│                                           │
│ 3. AES-OFB (256-bit key)              │
│    - 版本決定的 32-byte key             │
│    - 4-byte IV (來自 Hello Packet)     │
└────────────────────────────────────────┘
```

### MapleRef Wiki Opcode 核心概念

> "Actions that are defined by an identifier called an Operation Code (opcode for short), which is the first 2 bytes of the packet (a short)"

**Send vs Receive Opcode 區分**：
| 方向 | 說明 |
|------|------|
| **S→C (Send)** | 伺服器發送，客戶端接收 |
| **C→S (Recv)** | 客戶端發送，伺服器接收 |

**重要**：同一個 Opcode 數字在不同方向代表**完全不同**的含義！

### IDA 分析Opcode的方法 (RaGEZONE, 2024)

> "If you truly are looking to learn how the game works internally... I suggest you to learn how to use IDA"

**方法 1：SpiritIDAPlugin + MaplePacketPuller**
```
IDA Pro → 目標函數 → 右鍵 → SpiritIDAPlugin
→ 格式化 pseudocode → MaplePacketPuller → 封包結構
```

**方法 2：手動 IDA 分析**
```
IDA → 找到 CClientSocket::ProcessPacket
→ 追蹤 switch-case 分發
→ 分析各 Opcode 處理函數
```

**方法 3：MaplePacketPuller (Python CLI)**
```
IDA pseudocode → .txt → MaplePacketPuller → 封包結構輸出
```

### NPC 腳本核心結構

```javascript
var status = 0;

function start() {
    cm.sendNext("歡迎來到我的商店！");
}

function action(mode, type, selection) {
    if (mode == -1 || mode == 0) {
        cm.dispose();
        return;
    }
    if (mode == 1) status++;
    
    switch(status) {
        case 0:
            cm.sendSimple("選擇:\n #b#L0#購買#l\n #L1#離開#l");
            break;
        case 1:
            if (selection == 0) {
                cm.openShop(1);
            }
            cm.dispose();
            break;
    }
}
```

### NPC 腳本最佳實踐

| 做法 | 說明 |

| ✅ 使用 `cm.` 函數 | 普遍支援於所有音源 |

## 🔑 核心學習要點

### 登入器/客戶端修改
1. v83 是客戶端修改的最佳起點版本
2. STREDIT + Hex Editor 可完成大部分 IP 修改需求
3. IDA/x64dbg 是進階修改必備工具
4. localhost 版本需與伺服器 Opcode + AES Key 匹配

### 抓包分析
1. **第一個封包永遠是明文** — Hello Packet 是分析起點
2. Maple_Pshark 可直接 Hook 取得明文，無需解密
3. Wireshark + MaplePacketPuller + MapleRef Wiki 是經典工作流
4. v83 三層加密：MapleCrypto → Shanda → AES-OFB

### Opcode結構
1. v83 Opcode **幾乎全部被映射**，無需重新追蹤
2. Cosmic 是最完整、最活躍的 Opcode 參考來源
3. IDA 分析價值在於**客戶端修改**而非伺服器開發
4. 同一 Opcode 數字在 S→C 和 C→S 方向代表**完全不同**含義

### WZ修改
1. HaRepacker-resurrected 是現代 WZ 編輯首選
2. WzImg-MCP-Server 可讓 AI 直接分析/修改 WZ
3. NPC.wz 中的 TYPE 欄位決定客戶端行為
4. NX 格式比 WZ 更快更小，但需要轉換

### NPC腳本編寫
1. `cm.` 函數是跨音源通用的標準介面
2. Cosmic/IntransigentMS 使用 Nashorn + ES6，HeavenMS 使用 Rhino
3. 避免 `importPackage`，使用 `var` 而非 Java types
4. Dynamic Shop 腳本可實現免重啟商店更新
